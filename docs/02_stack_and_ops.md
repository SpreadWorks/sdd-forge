# 02. 技術スタックと運用

## 説明

### 依存パッケージ

<!-- @text: プロジェクトの依存パッケージ管理方法を説明してください。 -->

`package.json` の `dependencies` および `devDependencies` はともに空であり、外部 npm パッケージへの依存は一切ありません。すべての機能は Node.js 標準ライブラリのみで実装されています。そのため、`npm install` 後に追加モジュールが導入されることはなく、インストール環境を問わず同一の動作が保証されます。パッケージの公開対象は `files` フィールドで `src/` に限定されており、ソースコード以外のファイルは npm レジストリに含まれません。

### デプロイフロー

<!-- @text: デプロイの手順とフローを説明してください。 -->

リリースは npm レジストリへの公開によって行います。公開前に `npm pack --dry-run` を実行し、含まれるファイルと機密情報の有無を確認してください。確認後、`npm publish --tag alpha` でプレリリース版を公開します。`--tag alpha` では `latest` タグが更新されないため、npmjs.com のパッケージページへ反映するには続けて `npm dist-tag add sdd-forge@<version> latest` を実行する必要があります。一度公開したバージョン番号は再利用できないため、`package.json` のバージョンは公開前に必ず更新してください。リリース操作はユーザーが明示的に指示した場合のみ実行し、自己判断で行ってはなりません。

### 運用フロー

<!-- @text: 運用手順を説明してください。 -->

ドキュメントの生成・更新は `sdd-forge build` で一括実行できます。このコマンドは `scan → init → data → text → readme` の順でパイプラインを処理します。個別に実行する場合は、まず `sdd-forge scan` でソースコードを解析して `analysis.json` を生成し、`sdd-forge data` で `@data` ディレクティブを、`sdd-forge text` で `@text` ディレクティブをそれぞれ解決します。ドキュメントの品質確認は `sdd-forge review` で行い、PASS するまで `sdd-forge forge` による改善を繰り返します。機能追加・修正時は SDD フローに従い、`sdd-forge spec` で spec を作成してから実装を開始してください。テストは `npm run test` で実行します。
