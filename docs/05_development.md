# 05. 開発・テスト・配布

## 説明

<!-- @text: この章の概要を1〜2文で記述してください。ローカル開発環境のセットアップ・テスト戦略・リリースフローを踏まえること。 -->

本章では sdd-forge パッケージの開発に必要なローカル環境構築（`npm link` によるビルドレスな開発サイクル）、ブランチ運用・コミット規約、および `development` ブランチから `main` への squash merge を経て `npm publish` で公開するリリースフローを説明する。
現時点ではテスト自動化フレームワークを持たず、`npm link` 経由の手動実行による動作確認が検証の主手段となっている。


## 内容

### ローカル開発環境のセットアップ

\`\`\`bash
git clone <repository>
cd <project>
npm link          # グローバルコマンドとして登録
<command> help    # 動作確認
\`\`\`

<!-- @text: 開発中にツール自身を実行する方法と、変更が即座に反映される仕組みを説明してください。 -->

`npm link` を実行すると、グローバルの `sdd-forge` コマンドがリポジトリの `src/bin/sdd-forge.js` へシンボリックリンクされる。
パッケージは `"type": "module"` で構成されており、ビルドステップを持たない。Node.js が ES モジュールをソースファイルから直接読み込む構成のため、ソースを編集した内容は次回コマンド実行時に即座に反映される。
トランスパイルやバンドルの手順は不要であり、エディタで保存後にそのままコマンドを再実行して動作確認できる。


### ブランチ戦略とコミット規約

<!-- @text: ブランチ運用（main/development の役割・squash merge 方針）とコミットメッセージの形式を説明してください。 -->

`main` ブランチはリリース済みの安定版のみを管理し、直接コミットは行わない。
日常的な開発は `development` ブランチ上で行い、リリース時に `development` から `main` へ squash merge で取り込む。
これにより `main` のコミット履歴は各リリース単位で1コミットに集約される。

コミットメッセージは以下の形式で英語で記述する：

\`\`\`
<type>: <subject>
\`\`\`

`type` には `feat`（機能追加）、`fix`（バグ修正）、`docs`（ドキュメント更新）、`refactor`（リファクタリング）、`chore`（雑務・設定変更）、`test`（テスト）を使用する。
日本語のコミットメッセージは使用しない。


### SDD ワークフロー

| コマンド | 説明 |
| --- | --- |
| `sdd-forge spec --title "..."` | spec 初期化 |
| `sdd-forge gate --spec ...` | spec ゲート |
| `sdd-forge forge --prompt "..."` | docs 反復改善 |
| `sdd-forge review` | docs レビュー |

### テスト

<!-- @text: テスト戦略・使用フレームワーク・テストの実行方法を説明してください。フィクスチャの構成も含めること。 -->

現時点では自動テストフレームワークは導入されておらず、`package.json` にもテストスクリプトは定義されていない。テストファイル（`*.test.js`、`*.spec.js`）も存在しない。

動作検証は `npm link` でインストールしたコマンドを実際のプロジェクトディレクトリに対して実行する手動テストで行う。新機能や修正を加えた際は、対象サブコマンドを実際の解析対象リポジトリに対して実行し、出力結果を目視確認することが現在の検証手順となる。

将来的にテストを導入する場合は `test/` ディレクトリを作成し、フィクスチャとして最小構成の PHP-MVC プロジェクトを `test/fixtures/` 以下に配置する構成を想定する。フィクスチャには Controller・Model・Shell の最小サンプルファイルと `app/Config/routes.php` を含め、各アナライザの出力が期待 JSON と一致するかを Node.js 標準の `assert` モジュールで検証する形が想定される。


### リリースフロー

\`\`\`bash
npm version patch   # 0.1.0 → 0.1.1
npm version minor   # 0.1.0 → 0.2.0
npm publish         # npm レジストリに公開
\`\`\`

<!-- @text: development → main への squash merge からnpm publish までのリリース手順を説明してください。 -->

`development` ブランチでの変更が完了したら、GitHub 上で `development` → `main` への Pull Request を作成し、squash merge でマージする。
マージ後、ローカルで `main` ブランチをチェックアウトして `git pull origin main` で最新化する。
次に `npm version patch`（バグ修正）または `npm version minor`（機能追加）を実行する。このコマンドは `package.json` の `version` フィールドを更新し、`v0.1.1` 形式のタグコミットを自動生成する。
`git push origin main --tags` でコミットとバージョンタグをリモートに反映する。
公開前に `npm pack` を実行して配布物を確認する。`package.json` の `files` フィールドに `"src/"` のみが指定されているため、パック内容が `src/` 配下のファイルに限定されていることを目視確認する。
問題がなければ `npm publish` で npm レジストリに公開する。初回公開時は `npm login` で認証済みであること。


### 技術スタックと依存関係

<!-- @text: 使用言語・ランタイムバージョン要件・npm依存関係の方針（依存ゼロ等）を説明してください。 -->

実装言語は JavaScript（ES modules、`"type": "module"`）であり、ランタイムは Node.js 18.0.0 以上を要件とする（`engines.node: ">=18.0.0"`）。

`package.json` に `dependencies` フィールドは存在せず、外部 npm パッケージへの依存はゼロである。すべての機能は Node.js 組み込みモジュール（`fs`、`path`、`url`、`child_process`、`readline` 等）のみで実装する。この方針により、`npm install` なしでクローン直後から `npm link` して即座に利用できる構成を維持する。

配布物は `files` フィールドで `src/` 配下のみに限定しており、`npm publish` 時に不要なファイルが含まれない。
