# 05. 開発・テスト・配布

## 説明

<!-- {{text: この章の概要を1〜2文で記述してください。ローカル開発環境のセットアップ・テスト戦略・リリースフローを踏まえること。}} -->

本章では、sdd-forge 自身の開発に必要なローカル環境のセットアップ・テスト実行・npm パッケージのリリース手順を説明します。
sdd-forge はプロジェクトのドキュメント生成に SDD ワークフローを採用しており、機能追加・修正時はそのフローに沿って作業します。

## 内容

### ローカル開発環境のセットアップ

```bash
git clone <repository>
cd <project>
npm link          # グローバルコマンドとして登録
<command> help    # 動作確認
```

<!-- {{text: 開発中にツール自身を実行する方法と、変更が即座に反映される仕組みを説明してください。}} -->

`npm link` を実行すると、グローバルの `sdd-forge` コマンドがリポジトリの `src/sdd-forge.js` を直接参照するシンボリックリンクとして登録されます。
パッケージは ES Modules 形式（`"type": "module"`）で構成されており、Node.js はインポート時にファイルをキャッシュしないため、ソースコードの変更は追加のビルド手順なく次回コマンド実行時に即座に反映されます。
変更後は `sdd-forge help` や対象コマンドを直接実行して動作を確認してください。

### ブランチ戦略とコミット規約

<!-- {{text: ブランチ運用（main/development の役割・squash merge 方針）とコミットメッセージの形式を説明してください。}} -->

`main` ブランチはリリース済みの安定コードを管理し、`development` ブランチを日常の開発ベースとして使用します。
機能追加・修正は `development` から `feature/NNN-xxx` ブランチを切って作業し、完了後は squash merge で `development` に統合します。
`development` → `main` へのマージもリリース時に squash merge で行います。
コミットメッセージはすべて英語で記述し、sign-off 行や Co-authored-by トレーラーは含めません。

### SDD ワークフロー

| コマンド | 説明 |
| --- | --- |
| `sdd-forge spec --title "..."` | spec 初期化 |
| `sdd-forge gate --spec ...` | spec ゲート |
| `sdd-forge forge --prompt "..."` | docs 反復改善 |
| `sdd-forge review` | docs レビュー |

### テスト

<!-- {{text: テスト戦略・使用フレームワーク・テストの実行方法を説明してください。フィクスチャの構成も含めること。}} -->

テストには Node.js 組み込みテストランナー（`node --test`）を使用しており、外部テストフレームワークへの依存はありません。
テストファイルは `tests/` 以下に配置され、ソースコードのディレクトリ構造（`lib/`・`docs/`・`specs/`・`presets/`）に対応した形でサブディレクトリが分かれています。
テストの実行は以下のコマンドで行います。

```bash
npm test
# 内部: find tests -name '*.test.js' | xargs node --test
```

フィクスチャはテスト対象のコマンドに応じた一時ディレクトリや静的ファイルを各テストファイル内で管理しており、テスト実行後はクリーンアップされます。

### リリースフロー

```bash
npm version patch   # 0.1.0 → 0.1.1
npm version minor   # 0.1.0 → 0.2.0
npm publish         # npm レジストリに公開
```

<!-- {{text: development → main への squash merge からnpm publish までのリリース手順を説明してください。}} -->

リリース手順は以下の順で実行します。

1. `development` ブランチのテストがすべてパスしていることを確認します。
2. `development` → `main` を squash merge し、バージョンタグを付与します。
3. `npm pack --dry-run` でパッケージに含まれるファイルを確認し、機密情報や不要ファイルがないことをチェックします。
4. `npm publish --tag alpha` でプレリリースとして公開します（この時点では `latest` タグは更新されません）。
5. `npm dist-tag add sdd-forge@<version> latest` を実行して `latest` タグを更新し、npmjs.com のページに反映させます。

npm は一度公開したバージョン番号の再利用を許可しないため、公開前にバージョン番号を慎重に確認してください。
また、短時間に連続で publish するとレート制限によりブロックされる場合があるため、不必要な publish は避けてください。

### 技術スタックと依存関係

<!-- {{text: 使用言語・ランタイムバージョン要件・npm依存関係の方針（依存ゼロ等）を説明してください。}} -->

sdd-forge は JavaScript（ES Modules）で実装されており、Node.js 18.0.0 以上を動作要件としています。
外部 npm パッケージへの依存は一切なく、ファイル操作・子プロセス実行・HTTP 通信などすべての機能を Node.js 標準ライブラリのみで実装しています。
これにより、インストール時の依存解決エラーや脆弱性リスクを排除し、軽量なパッケージサイズを維持しています。
パッケージの公開対象ファイルは `src/` ディレクトリのみであり、テストコードや開発用設定ファイルは含まれません。
