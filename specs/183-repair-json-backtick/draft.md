# Draft: repair-json-backtick

**開発種別:** バグ修正 (BUG / robustness)

**目的:** AI 応答 JSON 修復ユーティリティを拡張し、value の開始引用符がバッククォートになっているケースを救う。これにより、retro 等の AI 応答パースが失敗してフェーズが skip される事象を防ぐ。

## 調査済み事項（事前探索）

- Issue #159 本文と原因（バッククォート開始 / 二重引用符閉じのハイブリッド）を確認
- 既存修復ユーティリティが対応する/しない壊れ方の範囲を確認
- バッククォート開始 value が現状救済対象外であることを確認
- 既存ユニットテストの存在と網羅範囲を確認

## 要件

### P1（必須）

- **R1** When `repairJson` がプロパティ値の開始位置でバッククォート (\`) を検出したとき, the system shall それを開き二重引用符として扱い、値を文字列としてパースして出力には二重引用符を書き込む
- **R2** When バッククォート開始 value の終端を検出するとき, the system shall 次に現れる二重引用符またはバッククォートのうち、直後（空白を除く）が JSON 構造区切り文字（`,` `}` `]` `:` または EOF）であるものを終端と判定し、出力に二重引用符を書き込む
- **R3** When 値本文中にバッククォートが含まれるとき, the system shall それらを通常の文字としてそのまま保持する（コード識別子表示の用途を壊さない）

### P2（重要）

- **R4** When 入力が JSON.parse でそのまま受理できる正常 JSON のとき, the system shall 既存ユニットテスト群が変更なしで全て pass する出力を返す
- **R5** When 開始・終端ともにバッククォートの値が含まれるとき, the system shall 双方を二重引用符に置換し、JSON.parse 可能な文字列を出力する

## スコープ外

- AI プロンプト側の強化（JSON 形式指示の厳格化）
- retro / review の再実行可能化（finalize cleanup の挙動変更）
- 配列 value の開始バッククォート（現実の発生事例なし。値全体ではなく構造解析で別途扱う）
- キー側のバッククォート開始（同上、現実事例なし）

## 既存機能への影響

- AI 応答 JSON 修復ユーティリティを呼ぶ既存パスは救済範囲が広がるだけで、正常 JSON の挙動は不変
- 既存ユニットテストは影響なし（既存ケースは現状通り動作）

## 制約

- 外部依存追加禁止（Node.js 組み込みのみ）
- alpha 版ポリシー: 後方互換シム不要、修復ユーティリティの API シグネチャは変えない

## 代替案

- **正規表現で全置換**: `:` 直後のバッククォート → `"` に単純置換。却下: 値本文中のバッククォート（`` `code` ``）も誤って置換するリスク
- **AI プロンプトで形式指示を強化**: 別アプローチ。完全には防げないため、parser 側の防衛が依然必要
- **JSON5 等の代替パーサー導入**: 外部依存禁止に反する

## テスト戦略

- **種別**: unit（既存の `repairJson` ユニットテストに追加）
- **検証対象**:
  1. R1/R2/R5: バッククォート開始 value（hybrid: open backtick / close double-quote、および open backtick / close backtick）が修復され `JSON.parse` 可能になる
  2. R3: 値本文に backtick を含む文字列がそのまま保持される
  3. R4: 既存の正常 JSON ケースで出力が変わらない（回帰なし）
- **配置判定**: 「将来変更で壊れた場合は常にバグ」→ YES。公式 `tests/` 配下に追加

## 将来拡張

- 同様の AI 出力癖（シングルクォート開始など）に対する追加修復ルートを最小差分で拡張可能
- backtick 修復ロジックは将来 review JSON など他のパース失敗にも自動的に適用される
