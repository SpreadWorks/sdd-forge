**開発種別:** Enhancement（既存フローへのステップ追加）

**目的:** flow-impl の実装着手前に、spec の各要件ごとの設計方針をユーザーが確認・承認できるステップを追加し、アーキテクチャ上の問題（引数過多・既存コード未活用・不適切なデザインパターン等）をコーディング前に発見できるようにする。

## Q&A

**Q: 変更対象はどのレイヤー？**
A: `src/templates/skills/sdd-forge.flow-impl/SKILL.md` のみ。CLI コマンドは作らない。

**Q: 方針確認の結果（approach.md）をファイルに保存するか？**
A: 保存しない。フロー進捗記録（step ID）も追加しない。

**Q: 承認/差し戻しの構造は？**
A: SKILL.md の選択肢形式（`[1] この方針で進める [2] 変更したい`）で実現。差し戻し後は修正指示を受けて再提示。リトライ上限は3回。autoApprove 時は自動承認してスキップ。

**Q: 提示する内容の粒度は？**
A: ファイル名の列挙ではなく、spec の各要件ごとに「どういう設計で実装するか・何の既存コードを使うか・なぜその設計か」を示す。ユーザーは「引数過多ではないか」「既存コードを使うべきでは」「別のデザインパターンが適切では」といった観点でレビューできる。

## 要件

1. flow-impl のステップ1（implement）の冒頭で、コードを書く前に実装方針を提示する
2. 提示フォーマット: spec の各要件ごとに「方針・使う既存コード・設計判断」を記載
3. ユーザーが `[1] 進める [2] 変更したい` で承認/差し戻しを選択できる
4. 差し戻し時: 修正指示を受けて方針を更新し再提示。リトライ上限3回
5. autoApprove が true の場合: 方針を提示した上で自動承認し即座に次に進む
6. 承認後にコーディングを開始する

## スコープ外

- `sdd-forge flow run approach` CLI コマンドの新設
- approach.md ファイルへの保存
- step ID (`approach-confirm`) の phases.js への追加

## 影響する既存機能

- flow-impl ステップ1の開始タイミングが変わる（コーディング前に確認フェーズが入る）
- autoApprove 時のフローに新しい自動処理が追加される（方針提示 → 自動承認 → コーディング）
