# アーキテクチャ議論メモ — 2026-03-09/10

## 発端

sdd-forge のアーキテクチャ全般に残っている構造的課題を洗い出した。
議論が深まり、個別課題の修正ではなく、プロダクトの思想・方向性の根本的な見直しに至った。

---

## Part 1: 個別の構造的課題（2026-03-09）

### 課題1: `docs.chapters()` の設計問題

**現象**: README の章テーブルの「概要」列に `<!-- {{/text}} -->` がそのまま表示される。

**原因の構造**:
1. テンプレートの各章に `## 説明` + `{{text}}` ディレクティブで概要を定義
2. `{{data: docs.chapters()}}` が各章ファイルを読み、`{{text}}` の生成結果を概要として抽出
3. build パイプラインは `data → text → readme` の順なので、`text` 未実行時は `{{text}}` が未埋め
4. AI の空レスポンス等で `{{text}}` が未埋めのまま残ると終了タグが漏れる

**本質的な問題**:
- `docs.chapters()` は `{{data}}`（決定論的データ）なのに `{{text}}`（AI 生成）の結果に依存
- データソース内でテンプレート的な振る舞い（マークダウンテーブル生成、ファイル読み込み、概要抽出）をしている
- **スクリプト内にテンプレートがある状態**
- `{{data}}` から `{{text}}` の結果を読んでいるのはこの箇所だけ

**バグ修正**: `docs.js` で `<!-- {{/text}} -->` のスキップ漏れを修正済み（コミット済み）。

### 課題2: データソース全体の責務混在

全32データソースファイル、約75メソッドが `toMarkdownTable()` でマークダウンテーブルを生成。`toMarkdownTable()` が `data-source.js` の基底クラスにあり、テンプレートの責務がデータソースに混入している。

テーブル系は実害が少ないが、`docs.chapters()` は他のディレクティブの生成結果を読みに行く依存がある点で特にまずい。

### 課題3: `review-parser.js` の CakePHP 依存

`patchGeneratedForMisses()` がファイルパス（`docs/08_controller_routes.md` 等）とカテゴリ名（`controllers`, `tables` 等）をハードコード。CakePHP 以外のプリセットでは forge の local モードが一切機能しない。

### 課題4: `modules` カテゴリの粒度

`genericScan` は既に削除済み。問題は「ハードコード」ではなく「粒度」。

- **webapp 系**: `controllers`, `models`, `shells`, `routes` と細かく分かれている。問題なし
- **node-cli 系**: 全ファイルが `modules` という単一カテゴリに入る。章との対応が取れない
- preset.json でフォルダベースの分割を定義するのは無理（プロジェクトごとに構造が違う）

## Part 2: 根本的な問題の発見（2026-03-10）

### AI がソースコードを読んでいない

build パイプライン全体を通して、**AI がソースコードの中身を一切読んでいない**ことが判明。

AI に渡されているのは:
1. **`projectContext`** — `setup` 時にユーザーが手入力した概要文
2. **analysis.json のサマリー** — scan で正規表現抽出した構造情報（ファイル数、クラス名、メソッド名）
3. **テンプレートのプロンプト** — 「認証方式を説明してください」等の指示
4. **周囲20行** — ディレクティブ前後のテンプレートの文脈

結果として、生成されるドキュメントは **「コントローラが15個あります」レベルの表面的な内容** に留まり、ビジネスロジックや設計意図は書けない。

### テンプレート穴埋めの粒度問題

各章の中に `{{text}}` ディレクティブが5〜10個あり、個別に AI に指示している。これにより:
- AI に十分なコンテキストを渡せない（断片的なプロンプト）
- ディレクティブ間の整合性が取れない（独立して生成）
- テンプレート作成の負荷が高い（プリセットごとに大量のディレクティブを設計）
- テンプレートに「認証方式を説明して」と書ける人は、自分で書いた方が早い

## Part 3: 思想の議論（2026-03-10）

### プロダクトの存在意義

**前提**: AI がほぼすべてのコーディングを行う時代において:
- コードレビューはしばらく残るが、いずれ AI が品質・セキュリティも担保する
- 人間に必要なのは「何が間違っているかを把握する能力」
- AI に聞くことはできるが、「何を聞くか」のガイドラインが必要 → それがドキュメント
- ソースコードが正だが、理解に時間がかかる
- AI が機械語を直接生成する時代になれば、人間が読めるソースコード自体がなくなる可能性
- **その時、人間がシステムを理解するための唯一の手段がドキュメントになる**

### テンプレートの役割の再定義

テンプレートは「穴埋めの枠」ではなく、**「人間がシステムを理解するために何を知る必要があるか」の定義**。

プリセットは「webapp ならこれを知る必要がある」「CLI ツールならこれを知る必要がある」という **知識体系の定義** になる。

### 多段パイプラインの是非

議論の変遷:
1. 「多段パイプラインは不要、1ステップで済む」→ 誤り
2. 「章ごとに同一処理を繰り返すだけ」→ 更新コストの観点が欠落
3. 「各ステップがあることで更新時のコストが下がる」→ 正しい
4. **最終的な結論**: パイプラインの段数自体は問題ではない。**各段が何をすべきか**の問題

### sdd-forge の本当の価値

**「テンプレート穴埋めツール」ではなく「コードとドキュメントの同期エンジン」であるべき。**

開発者が本当に欲しいのは:
- コードを変更したら、影響するドキュメント箇所を教えてくれる
- その箇所だけ正確に更新してくれる
- 更新内容がコードの実態と一致していることを保証してくれる

### 競合との差別化

Claude Code に「このプロジェクトのドキュメントを書いて」と言えば、ソースを全部読んで書いてくれる。構成はブレるが中身は正確。sdd-forge の価値が「構成の統一」だけなら、Claude Code にテンプレートを渡せば済む。

sdd-forge が勝つべき領域:
- **構造の統一と再現性** — テンプレートによる出力の安定
- **差分検知と部分更新** — コード変更時に影響箇所だけ再生成
- **コードとドキュメントの同期保証** — 乖離の検出と自動修正
- **知識体系の定義** — プロジェクトタイプごとに「何を知るべきか」を体系化

## Part 4: 今後の設計方針（確定 — 2026-03-10）

### 解決すべき核心

**AI がソースコードを読んで、意味のあるドキュメントを書ける仕組みを作る。**

### 設計判断（合意済み）

#### 1. 「関連ソース」をどう特定するか

- テンプレートのディレクティブが自然にマッピングを定義している（`{{data: controllers.list()}}` → controllers カテゴリ）
- `enrich` ステップで AI が全体像から各エントリーに章分類を付与
- **全体から個**へ。エントリーごとに個別に AI を呼ぶのではなく、全体像を AI に渡して一括で分類・要約する
- 規模に関わらず同一アーキテクチャ

#### 2. テンプレートの粒度

- **`{{text}}` ディレクティブの粒度は維持する**
- 構成の安定性が sdd-forge の価値。AI に段落構成を任せると生成ごとにブレる
- プリセットのテンプレートは汎用的な観点を定義（ドメイン知識不要）
- カスタマイズは `.sdd-forge/templates/` か、ディレクティブを消して手書き
- `{{text}}` は雑な指示でも AI が整えてくれる（校正ツールとしての価値）

#### 3. 差分更新の仕組み

- 全体 scan ではなく**差分 scan**
- 変更検知（git diff 等）→ 変更ファイルだけ analysis に追加/更新 → そのエントリーだけ enrich → 影響するカテゴリの対応章だけ text 再実行
- SDD フローなら spec に変更内容が記載済みなので、影響範囲は既知
- `forge --spec` がこの情報を活用

#### 4. パイプラインの再設計

**新パイプライン:**

```
scan    → 構造抽出（差分対応）
enrich  → AI で全体像から各エントリーに情報付与（★新規）
init    → enriched analysis + テンプレートから章組み立て
data    → 決定論的データ埋め込み
text    → light/deep 選択で AI テキスト生成（関連ソース付き）
readme  → README 生成
```

**text の 2 モード:**
- `light`: enriched analysis の情報を文章化（安い・速い）
- `deep`: 該当ソースコードを再読して詳細に書く（高い・正確）

**forge（差分更新）:**
```
差分検知 → 影響章特定 → その章だけ text 再実行
```

### enriched analysis のフォーマット（合意済み）

```json
{
  "modules": [{
    "name": "scanner.js",
    "methods": ["findFiles", "parseJS"],
    "summary": "ファイル探索とJS/PHP解析のユーティリティ",
    "detail": "findFiles はglob パターンでファイルを再帰探索し...",
    "chapter": "internal_design",
    "role": "lib"
  }]
}
```

- `summary`: 軽量コンテキスト（text light モードで使用）
- `detail`: ソースから読み取った詳細情報（間引かずそのまま保持）
- `chapter`: 対応する章の識別子
- `role`: エントリーの役割分類

### 既存の DataSource / `{{data}}` ディレクティブ

- スコープが大きいため**一旦は維持**
- enrich 導入後に段階的に見直す

### 移行戦略

- **一括リプレース**（A）を採用。段階的移行はしない

### 廃止

- `projectContext`（ユーザー手入力の概要文）— ソースコードから情報を取得するなら不要
- `summary.json` — enriched analysis の `summary` / `detail` フィールドが役割を吸収。章ごとにフィルタして渡すので全体の軽量版ファイルは不要

### 維持すべき既存の価値

- テンプレートによる構造の統一
- `{{text}}` ディレクティブによる構成の安定性
- プリセットによるプロジェクトタイプ別の知識体系
- review による品質チェック
- SDD フロー（spec → gate → implement → review）

## Part 5: プロダクト評価（2026-03-10）

### 現状（enrich 導入前）の評価

- analysis.json のメタデータだけで AI が書いたドキュメントは表面的。Claude Code に「ドキュメントを書いて」と言った方が中身は正確
- 現状の優位性は構成の安定性のみ。中身が空虚なら安定した構成に価値はない
- **結論: 現状では競合優位性が不十分**

### 新アーキテクチャ導入後の評価

**勝てる領域:**
1. **構成の安定性 + 中身の正確性** — Claude Code にない「毎回同じ構成」を正確な内容と両立
2. **差分更新** — 最大の武器。コード変更 → 影響章だけ更新を仕組みとして保証
3. **知識体系の定義** — プロジェクトタイプ別に「何を知るべきか」を体系化

**懸念と反論:**

| 懸念 | 反論 |
|---|---|
| AI に直接聞けば docs 不要では？ | docs は AI コーディングツールのガイドラインとしても機能する。AGENTS.md/CLAUDE.md に反映されれば、AI が既存設計から逸脱しないガードになる。個人開発でも価値あり |
| AI は「なぜそうしているか」を書けない | forge で人間が追記できる。ディレクティブ外の手書き内容は自動上書きされない。「what は AI、why は人間」で docs を育てられる |
| セットアップコストが高い | setup → build 一発で動く。本当のハードルはセットアップではなく「このプロダクトが何であるか」を理解すること。sdd-forge 自身の docs の品質が答えになるべき（ドッグフーディング） |

### docs の追加的価値: AI の行動制約

```
docs が既存設計を記述
  → AGENTS.md / CLAUDE.md に反映
  → AI コーディングツールがそれに従ってコードを書く
  → コードが既存パターンから逸脱しない
  → docs とコードの整合性が維持される
```

ドキュメントは「読むもの」だけでなく「AI の行動制約」として機能するフィードバックループ。

### ターゲット

| ターゲット | 価値 | 理由 |
|---|---|---|
| チーム開発（中〜大規模） | **高い** | オンボーディング、知識共有、ドキュメント鮮度維持 |
| OSS プロジェクト | **高い** | コントリビューター向けの一貫したドキュメント |
| 個人開発 | **中** | AI の行動制約としての価値（ガードレール） |
| コンプライアンス要件あり | **高い** | 監査用の構造化されたドキュメント |

### 総評

新アーキテクチャ（enrich + ソース読み込み + 差分更新）が実装されれば、**構成の安定性 × 差分更新** の組み合わせが真の差別化になる。CI に組み込んで「コード変更 → docs 自動追従」が確実に動くなら、使う価値がある。
