# Draft: 028 - SDD フローへのテスト駆動開発の導入

**Status**: Draft (議論中)
**Created**: 2026-03-09

## 背景・課題

現状の SDD フローでは、実装後にテストを作成している。このため:

- テストが「書いたコードの追認」になり、コードが動作することの確認に偏る
- コーディングのゴールが「テストを追加すればOK」にすり替わる
- 本来のゴールである「仕様に合致したものを作ること」が検証されにくい

## 検討した代替アプローチ

### A. ビルド実行 + 出力検証

実装後に `sdd-forge build` 等を実行し、生成物を自動検証する。

- 効果: 30〜40%（統合レベルのバグ検出）
- 実質的に E2E テストを書くことと同義

### B. spec と実装の照合チェック

spec の Requirements を1行ずつ読み、対応するコード変更を照合する。

- 効果: 15〜20%（仕様の取りこぼし防止）

### C. ユーザー検証シナリオ

spec に「これが動けば OK」という検証シナリオを記述する。

- 効果: 20〜30%（ユーザー期待とのズレ防止）
- 実質的に E2E テストを書くことと同義

### D. 影響範囲の事前宣言と事後検証

変更影響範囲を事前宣言し、宣言外の変更を警告する。

- 効果: 10〜15%（意図しない副作用の防止）

### E. タスク分解 + 段階的テスト

人間の開発手法に倣い、spec → タスク分解 → タスクごとに単体テスト → 結合テスト。

- 効果: タスク分解自体は AI が暗黙的に行っている
- coding agent の進化で不要になる可能性がある
- フローで強制するオーバーヘッドに見合うか疑問

### F. テスト手法の検討

| テスト手法 | AI 実装との相性 | Web アプリへの適用 |
|---|---|---|
| ブラックボックス | 高い。仕様ベースで実装バイアスが入りにくい | 適用可能 |
| ホワイトボックス | 低い。AI が実装とテストを同じ理解で書く | 適用可能 |
| スナップショット | 高い。AI の判断に依存しない | API は可能、DB/UI 依存は困難 |
| プロパティベース | 高い。性質の定義が仕様に近い | ビジネスロジック層は可能、Controller/View は困難 |
| ミューテーション | 高い。テスト自体の品質を検証できる | 適用可能 |
| ファズ | 中。エッジケース発見に有効 | 適用可能 |

**プロパティベーステストについて:**
- 「どんな入力でも成り立つ性質」を定義して検証する手法
- ファズ（ランダム入力生成）を内包する
- ユーザー確認が「この性質は成り立つべきか？」だけで済む
- 外部ライブラリ（`fast-check` 等）が必要だが、手動入力パターンで簡易的に代替可能

### 検討結果

- A, C は「E2E テストを書く」と同義であり、TDD の中に含められる
- E のタスク分解は AI が既に暗黙的に行っており、フローでの強制は過剰
- **最も効果が高いのは「テスト観点にユーザーのレビューが入ること」**
- タスク分解は AI の裁量に任せ、仕様とテスト観点だけユーザーが関与する方針とする
- テスト手法の選択はプロジェクトの種類と変更内容に応じて AI が判断する

## 提案: TDD フローの導入

### フロー変更

```
現在: spec → 承認 → gate → 実装 → テスト追加 → forge → review

提案: spec → 承認 → gate
        → テスト環境の確認
        → テスト観点を提示 → ユーザーが OK / 修正
        → テストコード生成（or AI 検証）+ fail 確認
        → Acceptance Criteria にテスト pass を追加
        → 実装（タスク分解は AI の裁量。テスト pass を目指す）
        → テスト pass 確認
        → forge → review
```

### テスト環境の自動判定

analysis.json（`sdd-forge scan` の出力）からテスト環境の有無を自動判定する。
ユーザーへの確認は不要。

判定ソース:
- `devDependencies` のテストフレームワーク（jest, mocha, vitest, phpunit 等）
- `scripts.test` の有無
- `tests/` ディレクトリの有無
- `composer.json` の `require-dev`（phpunit 等）

| 判定結果 | 対応 |
|---|---|
| テスト環境あり | 既存のフレームワーク・構成に合わせてテストコードを追加 |
| テスト環境なし | AI が spec の Requirements と実装の diff を照合して検証。テストコードは書かない |
| テスト環境の構築が必要な場合 | 別 spec として扱う（現在の機能 spec のスコープ外） |

> 現状の scan では `scripts.test` を取得していないため、対応が必要。

### テスト確認の UX

ユーザーの判断コストを最小にする。提示は以下のレベルで行う:

```
テストを作成します。種別を選んでください:
1. ユニットテスト
2. E2Eテスト
3. 両方
4. 任せる

---

以下のテストを実施します:

1. 終了タグがある場合、開始〜終了間が生成領域として認識される
2. 終了タグがない場合、エラーとして扱われる
3. strip 処理で生成済みコンテンツが除去される

よろしいですか？
```

- テストコードや入出力の詳細はこの時点では見せない
- ユーザーは「このゴールに対してこのテスト観点で十分か」だけ判断する

### 設計判断

| 項目 | 決定 | 理由 |
|---|---|---|
| テストケースの粒度 | Acceptance Criteria を具体的なテストケースに分解 | |
| テストケースの管理場所 | spec.md 内（別ファイルでも可） | |
| テストを書くタイミング | gate 通過後、実装前 | ゴールのすり替え防止 |
| テスト種別の選択 | ユーザーに毎回確認（4択） | |
| テスト観点のレビュー | ユーザーが確認・修正 | 最も品質に効く部分 |
| テストコードの確認 | 不要（AI が生成・実行） | ユーザーの負担軽減 |
| spec 承認とテスト確認 | 分離する（別ステップ） | 判断の集中 |
| タスク分解 | AI の裁量に任せる | 既に暗黙的に実行されている。agent 進化で不要になる |
| テスト手法の選択 | AI がプロジェクト種別と変更内容から判断 | Web アプリと CLI で適用可能な手法が異なる |
| テスト環境がない場合 | AI による spec-実装照合チェック | テストコードなしでも最低限の検証は行う |
| テスト環境の構築 | 別 spec として扱う | 現在の機能 spec のスコープに含めない |

## 効果の見積もり

| 要素 | 改善度 | 備考 |
|---|---|---|
| テスト順序の変更（先に書く） | 5〜10% | AI が書く場合、順序だけでは効果が限定的 |
| テスト観点のユーザーレビュー | 20〜30% | 現状にない要素。最も効果が大きい |
| TDD フロー全体（テスト環境あり） | 25〜35% | |
| TDD フロー全体（テスト環境なし） | 15〜20% | AI 照合チェックのみ |

### AI 実装における TDD の限界

TDD の本質的価値は「実装者が仕様を正しく理解しているかをテストで検証する」こと。
AI が実装する場合、テストも実装も同じ AI の理解に基づくため、理解の誤りが両方に反映される。

品質向上の鍵は TDD の順序ではなく、**テスト観点にユーザーのレビューが入ること**。
coding agent が進化しても、このステップの価値は変わらない。

## 追加提案: スナップショットテスト（リグレッション検出）

### 背景

TDD フローの限界として「AI がテストも実装も書くので、理解の誤りが両方に反映される」問題がある。
これを補完するため、**AI の判断に依存しない検証手段**としてスナップショットテストを導入する。

### 考え方

過去の正常動作の出力を基準として保持し、コード変更後に同じ入力で再実行して差分を検知する。

```
固定入力（テンプレート + analysis.json）
  → scan: ソース解析結果のスナップショット
  → init: テンプレートマージ結果のスナップショット
  → data: {{data}} 解決結果のスナップショット
  → review: チェック結果のスナップショット
```

- `text`（AI 呼び出し）は非決定的なので除外し、deterministic な処理のみ対象とする
- 差分があれば検知 → 意図的な変更ならスナップショット更新、意図しない差分ならバグ

### 重要な前提: スナップショットの正しさは保証されない

スナップショットは「過去のある時点の出力」であり、その出力自体が正しいとは限らない。

対策:
- **スナップショット作成時にユーザーが承認する**: 初回または更新時に出力を確認してから保存する
- **差分ベースで判断する**: 「正しいか」ではなく「変わったか」を検知する。変わった場合にユーザーが判断する
- **スナップショットに信頼度を持たせない**: あくまでリグレッション検出ツールであり、正しさの証明ではない

つまりスナップショットテストの目的は:
- **「意図しない変更がないこと」の確認** — 正しさの保証ではない
- 差分が出たらユーザーに「この変更は意図的ですか？」と問う
- 意図的であればスナップショットを更新する

### 効果の見積もり

| 要素 | 改善度 | 備考 |
|---|---|---|
| スナップショットテスト単体 | 30〜50% | AI の判断に依存しない外部基準での検証 |
| TDD + スナップショット併用 | 40〜60% | テスト観点レビュー + リグレッション検出の組み合わせ |

TDD 単体（15〜20%）と比較して、スナップショットテストの方が効果が大きい可能性がある。
両方を組み合わせることで相互補完が期待できる。

### OSS ベンチマークの位置づけ

外部 OSS プロジェクトをベンチマークに使うアプローチも検討したが、これはテスト時ではなく
**spec 作成時に仕様の網羅性を担保する**ための参照として活用する方が適切。

- 多様なプロジェクト構造に対して仕様が網羅的か確認する
- 現時点では sdd-forge の外部ユーザーが少なく、活用は将来的な話

## 追加提案: draft（要件定義）フェーズの導入

### 背景

現状の SDD フローは、ユーザーの要望（要求）からいきなり spec（仕様書）を書いている。
本来のソフトウェア開発は:

```
要求定義（何がしたいか）→ 要件定義（何を作るか）→ 仕様書（どう作るか）
```

現状はこの「要件定義」をスキップしているため:
- spec の Q&A が AI によって自動生成され、ユーザーが確認していない判断が含まれる
- 完成した spec に対して修正指示を出すのが大変（「○○の△△がダメなので□□して」と長文が必要）
- spec を読んで「なんか違う」と感じるが、どこが違うか言語化しにくい

**「何を作るか」の合意がないまま「どう作るか」を書いているのが根本原因。**

### 改善: draft フェーズの追加

```
現在: 要求 → spec（仕様書）を一気に書く → ユーザーが修正指示（大変）

提案: 要求 → draft で要件を対話的に詰める → spec に清書
```

draft フェーズでは:
- AI が選択肢を提示 → ユーザーが番号で選ぶ
- AI が方針を提示 → ユーザーが「それ違う」と短く返す
- 疑問点は AI が 1 問ずつ聞く（まとめて聞かない、勝手に回答しない）
- 判断根拠（「なぜこのアプローチか」）も対話の中で自然に決まる

### spec との違い

| | draft（要件定義） | spec（仕様書） |
|---|---|---|
| 目的 | 何を作るか合意する | どう作るか定義する |
| 形式 | 対話的・非定型 | 定型フォーマット |
| Q&A | 実際の対話から生成 | draft の記録を転記 |
| ユーザーの負担 | 短い返答・番号選択 | 要約を確認するだけ |
| AI の自由裁量 | 低い（都度確認） | 高い（詳細は AI が書く） |

### フロー変更

SDD フロー開始時に以下の選択肢を提示する:

```
進め方を選んでください:
1. 要件を整理してから仕様書を作成する（対話で方針を詰めてから spec を作成します）
2. 仕様書を作成する（要件が明確な場合。従来通り spec から開始します）
```

- **1 を選んだ場合**: draft フェーズで対話的に要件を詰める → spec に清書
- **2 を選んだ場合**: 従来通り spec を直接作成

```
1 の場合: 要求 → draft（対話で要件を詰める）
            → spec に清書（draft の合意内容を反映）
            → 承認（要約の確認のみ。詳細は対話で決済済み）
            → gate → テスト → 実装

2 の場合: 要求 → spec → 承認 → gate → テスト → 実装（従来通り）
```

### ユーザーの負担の変化

| フェーズ | 現状 | draft 導入後 |
|---|---|---|
| 要望伝達 | 短い（「○○したい」） | 同じ |
| 方針決定 | spec を読んで修正指示（長文が必要） | 選択肢を選ぶ・短い返答 |
| spec 確認 | 斜め読みで承認（不安が残る） | 要約だけ確認（詳細は対話で決済済み） |

### draft の終了条件

spec の承認と同じ仕組みで、draft にもユーザー承認チェックを設ける:

```markdown
## Draft Confirmation
- [ ] User approved this draft
```

承認後に spec に清書する。draft.md は specs/ に残す（対話の記録として保持）。

### SDD フロー中の行動原則

SDD フローのすべてのステップで、**次の行動について必ずユーザーに確認する**。
AI が勝手に次のステップに進まない。

### 脱線防止ルール

draft 内で疑問が出た場合:

1. 1つの疑問に対して **1往復で回答** する
2. 解決しなければ Open Questions に記録して先に進む
3. Open Questions は spec 作成時または実装中に解消する

### 効果の見積もり

| 要素 | 改善度 | 備考 |
|---|---|---|
| draft フェーズ導入 | 20〜30% | 要件の合意精度が上がり、後続すべてに効く |
| 脱線防止ルール | 5〜10% | 議論の効率化 |

spec の精度が上がることで、後続のテスト・実装すべての品質が底上げされる。
TDD やスナップショットとは異なるレイヤーでの改善であり、**併用で効果が乗算的に効く**。

## 補足事項

### テスト観点の修正ループ

ユーザーがテスト観点に修正を求めた場合、承認されるまで反復する。終了条件はユーザーの承認のみ。

### 既存テストとの関係

新しいテストの追加により既存テストが壊れる可能性がある場合、ユーザーに確認する。
既存テストの修正・削除を AI が勝手に行わない。

### テストの保守

機能変更により過去のテストが不要・不整合になった場合:
- AI が該当テストを検出し、ユーザーに指摘する
- 修正・削除の判断はユーザーに委ねる
- 大規模なテストリファクタリングは別途ユーザーが判断するフェーズで対応する

## 全体の効果見積もり

### 個別の効果

| 改善要素 | 改善度 | 対象レイヤー |
|---|---|---|
| draft フェーズ導入（要件定義） | 20〜30% | 上流（仕様の精度） |
| 脱線防止ルール | 5〜10% | 上流（議論の効率） |
| テスト順序の変更（先に書く） | 5〜10% | 中流（テスト設計） |
| テスト観点のユーザーレビュー | 20〜30% | 中流（テスト設計） |
| スナップショットテスト | 30〜50% | 下流（リグレッション防止） |

### 全体の効果（併用時）

各レイヤーの改善は乗算的に効く。上流で仕様が正しくなれば、中流のテストも下流の検証も精度が上がる。

| 組み合わせ | 改善度 | 備考 |
|---|---|---|
| draft のみ | 25〜35% | 上流だけでも後続に波及 |
| draft + TDD | 40〜55% | 仕様精度 × テスト観点レビュー |
| draft + TDD + スナップショット | **55〜75%** | 3 レイヤーすべてカバー |
| TDD + スナップショット（draft なし） | 40〜60% | 仕様の誤解リスクが残る |

**テスト環境なしの場合**: 上記から 10〜15% 減（AI 照合チェックのみ）

### 実装方針

3 つの改善（draft フェーズ、TDD フロー、スナップショットテスト）は **1 つの spec として同時に実装する**。
理由: 分割するとコンテキストが圧縮され、改善間の繋がりが失われるため。
