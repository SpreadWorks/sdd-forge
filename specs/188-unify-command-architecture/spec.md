# Feature Specification: 188-unify-command-architecture

**Feature Branch**: `feature/188-unify-command-architecture`
**Created**: 2026-04-17
**Status**: Approved
**Input**: GitHub Issue #167 — flow / docs / check / metrics の 3 種類のアーキテクチャパターン統一

## Goal

`sdd-forge` CLI 内で共存する 3 種類のコマンド実行アーキテクチャ（flow 型 / docs 型 / bare 型）を 1 パターンに統合し、全コマンドが同一の実行機構（引数解析・共有コンテキスト注入・ライフサイクル・出力処理）を経由するようにする。

## Scope

1. 全 CLI コマンド（flow / docs / check / metrics 配下）が同一実行機構を経由する構造への移行。
2. 共有状態（root / config / logger / agent / paths / flowManager 等）を全コマンドが単一の共有コンテキスト経由で取得する構造への統一。
3. コマンドメタデータ（help・引数定義・遅延ロード・ライフサイクルフック）の単一ツリー宣言化と、各ディスパッチャが自ドメインのサブツリーのみをロードできる構造の実現。
4. 出力モード（JSON envelope / raw stdout）の 2 択を各コマンドのメタデータ宣言で切替可能にする仕組み。
5. 引数解析をディスパッチャ層に一元化し、コマンド本体は parse 済みの入力オブジェクトを受け取る構造への変更。
6. パイプライン内での内部コマンド呼び出し形式（例: `docs build` が scan / enrich / init 等を直列実行する経路）を共通契約に統一。
7. 上記移行に伴う旧実行経路・旧シンボルの除去と、恒久テスト / spec 固有テストの追加。

## Out of Scope

- コマンド本体のビジネスロジック変更（scan / enrich / build / gate 等、既存コマンドが実施している処理の内容そのもの）。
- CLI 外部インターフェースの意図的な変更（サブコマンド名・flag/option 名・出力内容 semantics・終了コード）。R8 に基づき原則として完全互換。
- Envelope クラスの再設計。
- Container が保持する既存サービス（logger / agent / flowManager / i18n 等）の追加・削除・再設計。
- config スキーマ変更。
- プリセット / テンプレート側の変更。

## Clarifications (Q&A)

### Q1: Issue #167 の意図確認
- **A:** 3 パターン統一の方針で本 spec に落とし、Phase 4 の横断的アーキテクチャ統一として実装する。
- **根拠:** Phase 4 定義文書 (`.tmp/1877.md`) に「3 アーキテクチャパターンの統一」が目標として明記。

### Q2: スコープ（3 タスクまとめるか分割するか）
- **A:** 本 spec で 3 タスク（docs 共通ライフサイクル化・check/metrics の Container 化・3 パターン統合）をまとめて実施する。
- **根拠:** 相互依存のため、分割すると中間状態で 4 パターン共存となり手戻りを生む。alpha ポリシー「後方互換コード不要」と整合。

### Q3: 統合先の出力形式（全 envelope 化か選択可能か）
- **A:** envelope (JSON) / raw (stdout 直書き) の 2 モードを宣言で切替可能にする。
- **根拠:** 既存の docs/check/metrics は progress bar・人間向け text・csv 等ヒューマン志向の出力が UX 上必要。JSON 一色への統合は CLI UX を破壊する。

### Q4: 共有コンテキストと呼び出し時入力の役割分担
- **A:** 共有コンテキストは実行中不変の共有状態（root / config / logger 等）、入力はその呼び出し固有の CLI 引数および局所 override 値（出力言語・出力先等）。
- **根拠:** 既存の flow 系コマンド基底クラスが同形の契約を採用済み。Container 設計原則「派生値は共有コンテキストに持たない」と整合。

### Q5: コマンドメタデータ配置
- **A:** 単一統合ツリー構造。各ディスパッチャは自ドメインのサブツリーのみをロード可能とする。
- **根拠:** Issue 目標「3 パターン → 1 パターン」への忠実度、および「繰り返しパターンは共通化」というコーディングルールとの整合。

### Q6: テスト戦略
- **A:** 共通実行機構に対する恒久テストと、本リファクタ固有の移行前後検証テストを、配置先を分けて管理する。
- **根拠:** プロジェクトのテスト配置ルール「将来の変更で壊れたら常にバグなら恒久配置、spec 限定なら spec 配下」に従う。

## Alternatives Considered

1. **全ドメインを JSON envelope 化**: docs/check/metrics の人間向け出力が破壊される。却下。
2. **dispatcher のみ共通化、メタデータはドメインごとに分離**: 実装パス 1 本化は可能だが、「お作法のブレ」が残る。Issue 目標への忠実度が下がる。却下。
3. **メタデータ宣言を廃止し、実行単位クラスに static プロパティとして吸収**: 遅延ロードの実装が複雑化し、エントリ数増加時の起動コストが懸念される。却下。
4. **Issue を 3 spec に段階分割**: 中間状態で 4 パターン共存を生む。alpha ポリシーと衝突。却下。

## Requirements

優先度は上位ほど高い。

### R1 (最優先): 単一実行機構
When 任意の CLI コマンドが呼び出されるとき、shall 全コマンドが同一の実行機構（同じ引数解析・同じ共有コンテキスト注入・同じライフサイクル・同じ出力処理）を経由する。ドメインごとに独立した実行経路は残存しない。

### R2: 単一コンテキスト取得
When コマンド実装が共有状態（実行ルート、設定、ロガー、エージェント、パス群、フロー管理等）を参照するとき、shall 単一の共有コンテキスト経由でのみ取得する。コマンド内部での共有状態再解決は、CLI エントリポイントを除き禁止する。

### R3: コマンドメタデータの単一宣言
When コマンドのメタデータ（help 文言、引数定義、遅延ロード手段、ライフサイクルフック）が定義されるとき、shall 単一の統合ツリー構造内で宣言される。When 各ディスパッチャが自ドメインのコマンドのみを解決すれば十分な場面では、shall 他ドメインのメタデータをロードせずに済む構造になっている。

### R4: ライフサイクルフックの共通提供
When コマンドが pre / post / onError を宣言しているとき、shall 共通実行機構が次の順で呼び出す:
1. 実行前に pre を 1 回呼ぶ。
2. 正常終了時は post のみを実行後に 1 回呼ぶ。
3. 例外発生時は onError のみを 1 回呼ぶ。

未宣言のフックは呼び出されない。ライフサイクルフックは全ドメインで等しく利用可能である。

### R5: 出力モードの二種化
When コマンドが実行されるとき、shall 出力モードは `envelope` または `raw` のいずれか 1 種類がメタデータ宣言で明示される。envelope モードは JSON envelope を stdout に出力し、raw モードはコマンド本体が直接 stdout/stderr に書き込む。既存 CLI のヒューマン志向出力と progress bar は維持される。

### R6: 引数解析の dispatcher 一元化
When CLI 引数が処理されるとき、shall 共通ディスパッチャ層がメタデータの引数宣言（flags / options / positional）に従って parse する。コマンド本体は parse 済みの入力オブジェクトを受け取る。

### R7: パイプライン内の呼び出し互換
When 1 コマンドが他のコマンドを内部オーケストレーションとして呼び出すとき（例: ビルドパイプライン）、shall 共通契約で呼び出し可能であり、呼び出しごとに局所 override（出力言語・出力先ディレクトリ等）を指定できる。

### R8: CLI 外部互換の維持
When 本リファクタの前後で同一の CLI 引数列でコマンドを実行したとき、shall 以下の観測可能な挙動が同一である:
1. 受理されるサブコマンド名および flag/option 名。
2. 正常系の stdout 出力内容。
3. エラー系の stderr 出力と終了コード。
4. ファイルシステム副作用（生成ファイル・配置パス）。

If 何らかの理由でこれらに差分が生じる場合、shall spec にその差分とマイグレーション方針を明示列挙する。

### R8a: 終了コード契約
When コマンドが正常終了するとき、shall プロセスは終了コード 0 を返す。When コマンドが異常終了（引数不正・想定外の例外・想定内の失敗で処理継続不可能な場合）するとき、shall プロセスは 0 以外の終了コードを返す（既存の `EXIT_ERROR` 契約を踏襲）。When 共通実行機構が例外を捕捉して envelope を出力するとき、shall 終了コードも失敗を示す 0 以外に設定される。

### R8b: エラーの可視化（握り潰し禁止）
When 共通実行機構が例外を捕捉するとき、shall 次のいずれかで必ず可視化する:
1. envelope モード: 例外を envelope の `errors` フィールドに記録して stdout に出力する。
2. raw モード: 例外メッセージを stderr に書き出す。
3. ライフサイクルフック内での失敗: 親エンベロープの警告として記録するか、stderr に明示ログする（既存 `tryUpdateStepStatus` 等の「ERR_MISSING_FILE のみ黙殺、他は再 throw」契約に従う）。

未知のエラーコードを無言で握り潰すことは禁止する。

### R9: 新コマンド追加の一貫手順
When 新規コマンドを追加するとき、shall 同一の 2 段階手順（実行本体の実装 + メタデータ宣言の追加）のみで完結する。ドメインによって追加手順が分岐する設計は許容しない。

### R10: 検証容易性
When 共通実行機構に対するユニットテストが作成されるとき、shall テストは子プロセスを起動せずに直接 import で実行できる。When テストが共有コンテキストまたはコマンドメタデータに依存するとき、shall テストコードはそれらをテストダブルで注入・差し替えできる。

### R11: リソース境界
When 本リファクタの実装が進められるとき、shall 新たな再帰処理・リトライ処理・一括読込処理は導入されない。When 既存コードに存在するリソース境界（例: ファイル件数上限の定数）に遭遇したとき、shall 移行前後でその境界値を不変に保つ。If 実装中に既存境界を意図せず変更したことが判明した場合、shall spec にその変更と理由を明示列挙する。

## Acceptance Criteria

### AC1 (R1, R2)
`sdd-forge flow ...` / `sdd-forge docs ...` / `sdd-forge check ...` / `sdd-forge metrics ...` を実行した結果として、以下を grep で静的検証したとき 0 件である:
- コマンド実装ファイル内での `repoRoot()` 直呼び（CLI エントリポイントおよび共有コンテキスト初期化モジュールを除く）。
- コマンド実装ファイル内での `loadConfig()` 直呼び（同上）。
- `export async function main(` / `export { main }` を含むコマンドファイル。

### AC2 (R1, R6)
ディスパッチャからコマンドへの引数渡しが `process.argv` 書き換え経由ではなく、parse 済みオブジェクトの引数渡しのみで行われる。

### AC3 (R3)
全コマンドのメタデータが 1 つのモジュール（ツリー構造）から解決可能であり、かつ各ディスパッチャは他ドメインのサブツリーを import せずに起動できる。

### AC4 (R4)
ライフサイクルフック (pre / post / onError) の実行順序と例外時の選択的呼び出しが、ユニットテストで検証される。

### AC5 (R5)
全コマンドが envelope / raw のいずれか 1 種類を宣言しており、宣言と実際の stdout 形式が一致することをユニットテストで検証される。

### AC6 (R7)
`sdd-forge docs build` パイプラインが共通契約経由で scan / enrich / init / data / text / readme / agents の内部コマンドを順次呼び出し、(a) single-lang モード、(b) multi-lang モード（default 言語 + 1 つ以上の非 default 言語）、(c) `--regenerate` モード の 3 パターンが、移行前と同じ stdout・生成ファイル・終了コードで完了する。

### AC7 (R8, R8a)
移行前後で以下が完全一致することを acceptance テストで検証する:
- `sdd-forge flow --help`, `sdd-forge docs --help`, `sdd-forge check --help`, `sdd-forge metrics --help` の stdout と終了コード。
- 正常系: `sdd-forge flow get status`, `sdd-forge flow run gate`, `sdd-forge docs build`, `sdd-forge check config`, `sdd-forge check freshness`, `sdd-forge check scan`, `sdd-forge metrics token` の stdout と終了コード 0。
- エラー系: 未知サブコマンド、必須引数不足、`config.json` 欠損時の stderr と 0 以外の終了コード。

If いずれかに差分があれば spec を更新しマイグレーション方針を明記する。

### AC8 (R9)
新コマンド追加手順が 1 種類に収束していることを、「指定された新コマンドを追加するチェックリスト」形式で spec 配下に記載する。

### AC9 (R10)
共通実行機構のユニットテストが、プロセス起動を伴わずに実行される。共有コンテキストとメタデータの注入がテストダブルで可能であることを、実際のテストケースで示す。

### AC10 (既存テスト)
既存の `npm test` 全テストが移行後もパスする。

## Open Questions

- [x] pre/post/onError 以外のフック（例: finally）を本リファクタで採用するか — 採用しない。現行 flow 型で `finally` スロットは予約のみで未使用。YAGNI に従い R4 は pre/post/onError のみとする。将来必要になれば別 spec で追加。
- [x] 出力モード `envelope` / `raw` の命名の妥当性 — 実装フェーズで最終化する。spec 層は概念として 2 モードの存在を要求する（R5）に留め、識別子名は実装フェーズの判断とする。
