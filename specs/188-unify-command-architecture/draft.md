# Draft: unify-command-architecture

**Development Type:** リファクタリング（非機能）

**Goal:** `sdd-forge` CLI 内で現在共存する 3 種類のコマンド実行アーキテクチャパターンを 1 パターンに統合し、全コマンド（flow / docs / check / metrics）が同一の実行機構で呼び出されるようにする。

## Background

Issue #167 (親タスク #1877 / Phase 4) で定義された横断的アーキテクチャ統一。Phase 1〜3 で Container 基盤と flow 型の実装機構は整備済みだが、コマンド実行パターンは 3 種類が共存している:

| パターン | 使用 | 実行形態 | ライフサイクル | 出力形式 |
|---|---|---|---|---|
| **flow 型** | `sdd-forge flow ...` | 宣言的 registry + 基底クラス + 統一 dispatcher | pre/post/onError hooks あり | JSON envelope |
| **docs 型** | `sdd-forge docs ...` | コマンド単位の main() 関数 + 動的 import | なし | 自由形式 |
| **bare 型** | `sdd-forge check/metrics ...` | コマンド単位の main() 関数 + 動的 import | なし | 自由形式 |

bare 型のコマンドは Container が初期化されているにも関わらず、コマンド内部でコンテキスト (root / config) を再解決する実装が残っている。

## Requirements

各要件は「When/If + shall」の形式で記述する。優先度は上位ほど高い。

**R1 (最優先): 単一実行機構**
When 任意の CLI コマンドが呼び出されるとき、shall 全コマンドが同一の実行機構（同じ引数解析・同じコンテキスト注入・同じライフサイクル・同じ出力処理）を経由する。ドメインごとに独立した実行経路は残存しない。

**R2: 単一コンテキスト取得**
コマンド実装が共有状態（実行ルート、設定、ロガー、エージェント、パス群、フロー管理等）を参照するとき、shall 単一の共有コンテキスト経由でのみ取得する。コマンド内部で共有状態を再解決する経路は、CLI エントリポイントを除き禁止する。

**R3: コマンドメタデータの単一宣言**
When コマンドのメタデータ（help 文言、引数定義、遅延ロード手段、ライフサイクルフック）が定義されるとき、shall 単一の統合された構造の中で宣言される。When CLI 起動時に自ドメインのコマンドのみを解決すれば十分な場面では、shall 他ドメインのメタデータをロードせずに済む構造になっている。

**R4: ライフサイクルフックの共通提供**
When コマンドが pre / post / onError を宣言しているとき、shall 共通実行機構が次の順で呼び出す: (1) pre を実行前に 1 回、(2) 正常終了時は post のみを実行後に 1 回、(3) 例外発生時は onError のみを 1 回。未宣言のフックは呼び出されない。ライフサイクルフックは全ドメインで等しく利用可能である。

**R5: 出力モードの二種化**
When コマンドが実行されるとき、shall 出力モードは `envelope`（JSON envelope）または `raw`（コマンドが直接 stdout に書き込む）のいずれか 1 種類が明示宣言される。既存 CLI の外部挙動（ヒューマンリーダブル出力、progress bar、終了コード）を維持する。

**R6: 引数解析の dispatcher 一元化**
When CLI 引数が処理されるとき、shall 共通 dispatcher が registry の引数宣言（flags / options / positional）に従って parse する。コマンド実装は parse 済みの入力オブジェクトを受け取る。

**R7: パイプライン内の呼び出し互換**
When 1 コマンドが他のコマンドを内部オーケストレーションとして呼び出すとき（例: ビルドパイプライン）、shall 共通契約で呼び出し可能であり、呼び出しごとに局所 override（出力言語・出力先ディレクトリ等）を指定できる。

**R8: CLI 外部互換の維持**
本リファクタの前後で、shall 次の観測可能な挙動は同一である: (1) 受理されるサブコマンド名および flag/option 名、(2) 正常系の stdout 出力内容、(3) エラー系の stderr 出力と終了コード、(4) ファイルシステム副作用（生成ファイル・配置パス）。If 何らかの理由でこれらに差分が生じる場合、shall spec にその差分とマイグレーション方針を明示列挙する。

**R9: 新コマンド追加の一貫手順**
When 新規コマンドを追加するとき、shall 同一の 2 段階手順（実行本体の実装 + メタデータ宣言の追加）のみで完結する。ドメインによって追加手順が分岐する設計は許容しない。

**R10: 検証容易性**
統合後の共通実行機構は、shall 実プロセスを起動せずに独立ユニットテストから実行できる。共有コンテキストおよびコマンドメタデータは、テストから注入および差し替え可能である。

## Out of Scope

- コマンド本体のビジネスロジック変更（scan/enrich/build/gate 等の内部処理）
- CLI 外部インターフェースの意図的な変更（すべて R8 に従う）
- Envelope クラスの再設計
- Container が保持するサービス群（logger / agent / flowManager 等）の追加・変更
- config スキーマ変更

## Q&A

Q&A のすべての回答は、以下のいずれかに根拠を紐付ける:
- (a) 既存の設計分析ドキュメント
- (b) プロジェクトガードレール / コーディングルール
- (c) 既存のコードパターン

### Q1: Issue #167 の意図確認
**A1:** [1] 3 パターン統一の方針で本 spec に落とす。
**根拠:** (a) Phase 4 定義文書に「3 アーキテクチャパターンの統一」が目標として明記されている。

### Q2: スコープ（3 タスクまとめるか分割するか）
**A2:** [1] 3 タスクとも本 spec でまとめて実施。
**根拠:** (a) 親タスクの分析結果から 3 指摘が相互依存であることが判明している。(b) alpha ポリシー「後方互換コード不要」に従うと、中間状態で 4 パターン共存は手戻りを生むのみ。

### Q3: 統合先の出力形式（全 envelope 化か選択可能か）
**A3:** [1] envelope / raw の 2 モードを明示宣言可能にする。
**根拠:** (c) 既存コマンドは用途ごとに異なる出力（progress bar / 人間向け text / csv など）を返しており、JSON 一色への統合は CLI UX を破壊する。(b) コーディングルール「シンプルなインターフェースに十分な実装を隠す」に沿い、出力モードを宣言で切替可能にする。

### Q4: 共有コンテキストと呼び出し時入力の役割分担
**A4:** [1] 共有コンテキストは実行中不変、入力は呼び出しごとの引数および局所 override 値。
**根拠:** (c) 既存のコマンド基底クラスが同形の契約を既に採用している。(a) Container 設計分析で「派生値は共有コンテキストに持たない、由来を明示する」方針が確立されている。

### Q5: コマンドメタデータ配置（ドメイン分離 vs 単一統合）
**A5:** [1] 単一統合ツリー構造（各 dispatcher は自サブツリーのみ参照）。
**根拠:** (a) Issue 目標「3 パターン → 1 パターン」への忠実度。(b) コーディングルール「繰り返しパターンは共通ヘルパーに抽出」に沿い、同形のメタデータ宣言を複数ファイルに分散させない。

### Q6: テスト戦略
**A6:** [1] 共通実行機構に対する恒久的なユニットテストと、本リファクタ固有の検証（移行前後の挙動一致）とを、配置先を分けて管理する。
**根拠:** (b) テスト配置ルール「将来の変更で壊れたら常にバグは恒久配置、この spec 限定は spec 配下」に従う。

## Impact on Existing Features

- **flow コマンド群**: 共通実行機構側に統合されるが、CLI 外部挙動は完全互換。
- **docs コマンド群**: ドメイン固有の実行経路が共通機構に置換される。ビルドパイプラインの内部呼び出し形式が共通契約に変わるが最終出力は完全互換。
- **check/metrics コマンド群**: コマンド内部の共有状態再解決が共有コンテキスト経由に置換される。外部挙動は完全互換。
- **共有サービス群**: Container が保持する既存サービス（ロガー・エージェント・フロー管理等）の実装は変更なし。
- **テンプレート / プリセット**: 変更なし。
- **CLI 外部インターフェース**: 全サブコマンド名・flag/option 名・出力内容・終了コードは維持。If 意図せぬ不整合が発見された場合、shall spec に明示列挙する (R8)。

## Constraints

- **alpha ポリシー**: 後方互換コードを残さない。旧パターンの実行経路・シンボルは移行完了時に削除する。
- **外部依存なし**: Node.js 組み込みモジュールのみ使用。
- **OOP による型表現**: 実行単位は振る舞いと不変条件を型自身に所属させる形で表現する。
- **`src/` 固有情報禁止**: プロジェクト固有値を埋めない。
- **深いモジュール**: ディスパッチャ本体は薄く、共通実行機構が責務を十分吸収する。

## Alternatives Considered

1. **全ドメインを JSON envelope 化**
   - 却下理由: `docs build` progress bar、`check config` text、`metrics token` csv など人間向け出力形態が破壊される。CLI UX 互換を損なう。
2. **dispatcher のみ共通化、registry はドメインごとに分離**
   - 却下理由: 実装パス 1 本化は達成できるが、「お作法のブレ」が構造的に排除されない。Issue #167「1 パターン統一」への忠実度が下がる。
3. **registry 不要 / Command クラスに static メタデータ吸収**
   - 却下理由: 遅延 import（起動コスト抑制）の実装が複雑化。flow だけで 20+ エントリあり起動負荷の増大リスク。
4. **Issue を 3 spec に段階分割**
   - 却下理由: 相互依存のため中間状態で 4 パターン共存状態が発生。alpha ポリシー「旧フォーマット保持禁止」と衝突し手戻りを生む。

## Future Extensibility

- R9 により新コマンド追加は一貫手順 1 種類に収束する。
- 新ドメイン追加（将来 `sdd-forge <new-domain> ...` の追加）はメタデータサブツリー 1 件とディスパッチ配線のみで対応可能。
- R4 により横断的関心事（メトリクス記録・監査ログ等）をフック層で追加できる基盤になる。
- R3 の単一メタデータ宣言により、全エントリに対する静的検査（help 存在・引数仕様妥当性・キー重複検出）が実行可能になる。
