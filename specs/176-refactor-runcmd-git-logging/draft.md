# Draft: refactor runCmd git logging

**開発種別:** リファクタリング（バグ修正を伴う）
**目的:** 汎用コマンド実行ヘルパー内に埋め込まれた git ロギングが worktree 内で無限再帰を引き起こす問題を、設計を見直して解消する。git ログ記録は維持する。

## 背景

現状、汎用コマンド実行ヘルパー内の git ロギング呼び出しがコメントアウトされており、git コマンドのログがまったく取られていない状態。

再帰経路:
- 汎用ヘルパーが git を実行 → ロガーが git レコードを記録
- ロガーがログ出力先解決のためメインリポジトリのパスを取得（worktree 対策）
- メインリポジトリのパス取得は内部的に汎用ヘルパー経由で git を実行
- → 再帰

## 用語定義

- **業務 git 操作**: ユーザー要求や SDD フローを進めるために実行される git コマンド。具体的には commit, push, diff, branch, checkout, merge, worktree 操作, log, status, rev-list, add 等、ロガーから見える範囲で実行されるすべての git コマンドを指す。
- **基盤 git 操作**: ロガー自身がログ出力先パスを解決するために実行する内部 git コマンド。リポジトリトップレベル取得と共有 git ディレクトリ取得の 2 種類のみを指す。

## Q&A

### Q1: 設計方針
- **Decision**: 業務 git 操作と基盤 git 操作を別経路で扱う設計に変更する。業務 git 操作はロギング対象、基盤 git 操作はロギング対象外とする。
- **根拠**:
  - **ガードレール**: SRP（汎用コマンド実行ヘルパーが特定コマンド種別を知るのは責務違反）。
  - **既存コード**: git 横断ヘルパーモジュールが既に集約地として存在しており、業務 git 操作の入口として自然に拡張できる。

### Q2: 0003（トークン削減/コマンドプロキシ）との関係
- **Decision**: 0003 の本実装は本 spec に含めない。本 spec の成果物は 0003 の前哨として位置付け、将来 0003 を実装する際に破棄せず拡張のみで対応できる構造を残す。
- **根拠**:
  - **既存コード**: 0003 はボードに ideas として登録された未確定の構想であり、本 spec とは独立した spec として実装すべき。

### Q3: 置換スコープ
- **Decision**: 基盤 git 操作のみを例外として現行の経路を維持し、それ以外の業務 git 操作はすべて新経路に統一する。
- **MUST**: 例外箇所には除外理由のコメントを残す。
- **根拠**:
  - **ガードレール**: 「Complete Context」— 例外の境界とその理由を spec 内に明示する。
  - **既存コード**: 基盤 git 操作の 2 種類はロガーがログ出力先解決に使う既存の構造で、これらをロギング対象にすると再帰が必ず発生する。

### Q4: テスト戦略
- **Decision**: 新しい git 経路の単体テスト、および worktree 内での再帰回帰テストを永続テスト領域に追加する。
- **根拠**:
  - **既存コード**: 永続テスト領域に process / log / git 系のテストが既に集約されている。
  - **ガードレール**: 再帰回帰テストは将来同種バグを防ぐ普遍的価値があり spec 個別の関心事ではない。

## Requirements（優先順位順）

### P0（必須）
- **R1**: When 業務 git 操作（用語定義参照）が実行されたとき, the system shall コマンド・終了コード・stderr を含む git レコードをログに記録する。
- **R2**: When 業務 git 操作が worktree 内で実行されたとき, the system shall 再帰なくログを記録できる。
- **R3**: When 基盤 git 操作（用語定義参照）が実行されたとき, the system shall そのコマンドをロギング対象外とし、除外理由を示すコメントを残す。

### P1（強く推奨）
- **R4**: If 業務 git 操作の実行が失敗したとき, the system shall 失敗時の終了コードと stderr もログに記録する。

## スコープ外

- 0003（トークン削減/コマンドプロキシ）の本実装。
- gh コマンドのロギング体系化。
- ログ形式の変更（既存ログスキーマは維持）。

## 既存機能への影響

- **影響あり**:
  - git 横断ヘルパーモジュール: 内部経路が変わるが戻り値の形は維持され、呼び出し元には透過。
  - フロー系の git 呼び出し箇所: 経路の置換のみで挙動は変わらない。
  - 汎用コマンド実行ヘルパー: git 専用ロジック（コメントアウトされた TODO 含む）を削除。
- **影響なし**:
  - ロガー API の外部シグネチャ。
  - 既存ログファイル形式。
  - 非 git コマンド（gh, npm, claude 等）の挙動。
