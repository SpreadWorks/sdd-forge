# Draft: プリフィックスコメントによる構造化コメントシステム

**開発種別:** 機能追加（ENHANCE）
**目的:** コード中のコメントに `WHY:` 等のプリフィックスを付与して構造化し、minify で保持することで、text 生成 AI が明示された意図に基づいた高品質なドキュメントを書けるようにする。

## 背景

- Issue #157（d966 として board からドラフト化）
- `sdd-forge docs text` は minified コードを AI に渡してドキュメントを生成する。
- 現行の minify は単一行コメントを一律削除するため、設計判断の意図が text 生成時に失われる。

## 事前探索

本ドラフト作成前に以下を確認した。

- **既存実装**: `document` プリセットの guardrail には既に「非自明な判断には why コメントを書く」旨の記事（`comment-intent-not-logic`）が存在する。各言語ハンドラ（JS/PHP/Python/YAML）の minify は単一行コメントを一律削除している（ブロックコメントも同様に削除）。
- **既存パターン**: guardrail 記事は統一された構造（id / title / body / meta.phase）で記述されている。minify は言語ごとのハンドラに分離されている。
- **プロジェクト方針**: alpha 版ポリシー（後方互換コードを書かない）、YAGNI（過剰な防御コードを書かない）、`src/` に特定プロジェクト固有情報を含めない。

## スコープ

1. `document` プリセットの guardrail にプリフィックス付きコメント記述ルールを MUST 表現で追加。
2. JS / PHP / Python / YAML 各言語ハンドラの minify がプリフィックス付き単一行コメントを保持するように変更。

## 除外

- scan / DataSource / テンプレート変更
- ファイル役割タグ（`@role:` 等）
- `analysis.json` への格納
- 既存コードへの一括追加（段階的改善として、今後 SDD フロー経由で実装されたコードに対して AI が自然に付与する）

## Requirements

優先順位は P1（必須）> P2（望ましい）。

- **R1 (P1):** When minify が単一行コメントを処理する際、If そのコメントが `WHY:` / `HACK:` / `SECURITY:` のいずれかで始まる（大文字・コロン直後）ならば、minify は shall そのコメント行を保持する。
- **R2 (P1):** When minify が単一行コメントを処理する際、If プリフィックスが上記 3 種のいずれにも完全一致しない（小文字、書式違い、未定義プリフィックスを含む）ならば、minify は shall そのコメント行を削除する（現行挙動の継続）。
- **R3 (P1):** When コメントがコード末尾に付与されている場合（例: `foo(); // WHY: ...`）、minify は shall プリフィックスの有無に関わらず当該コメント部分を削除する。
- **R4 (P1):** When minify がプリフィックス判定を行う際、対象言語ハンドラは shall JS / PHP / Python / YAML の 4 つとし、それぞれの単一行コメント構文（JS: `//`、PHP: `//` および `#`、Python: `#`、YAML: `#`）に対して同一のプリフィックス集合を適用する。
- **R5 (P1):** When AI が SDD フロー経由で実装を行う際、`document` プリセットの guardrail は shall プリフィックス付きコメントの記述を MUST 表現で指示する。
- **R6 (P2):** When AI がプリフィックス付きコメントを記述する際、コメント本文は shall `config.docs.defaultLanguage` で指定された自然言語で書かれる（guardrail にて指示）。
- **R7 (P2):** When minify がブロックコメント（`/** */` など複数行コメント構文）を処理する際、minify は shall 現行の挙動を維持する（本変更のスコープ外）。

## Acceptance Criteria

- プリフィックス付き単一行コメントを含むサンプル入力を各言語ハンドラの minify に通したとき、当該コメント行が保持される（ユニットテストで検証）。
- プリフィックスなし・書式違いのコメントが削除される（ユニットテストで検証）。
- コード末尾コメントが削除される（ユニットテストで検証）。
- guardrail 記事 `comment-intent-not-logic` の body に `WHY:` `HACK:` `SECURITY:` の 3 プリフィックスが MUST 表現で明記されている。

## Q&A

### Q1. 意図確認

**推奨:** Issue #157 の内容どおり実施する。
**根拠種別:** Issue 本文（docs/）
**選択:** [1] はい（2026-04-17）

### Q2. プリフィックスセット

**推奨:** 中間 3 種 `WHY:` / `HACK:` / `SECURITY:` を採用する。
**根拠種別:** guardrail 原則（YAGNI）+ 既存コードパターン（guardrail の簡素さ）
**選択と理由:** [C] 中間 3 種。`TODO:` は開発者向けでドキュメントに載せたくない、`NOTE:` は乱用リスク。少なめから始め、運用実績で追加判断する（段階的改善のため既存コード移行コスト無し）。

### Q3. 検出ルール・対象言語

**推奨:** 大文字+コロン厳格書式、単一行行頭のみ、JS/PHP/Python/YAML の 4 言語、末尾コメント対象外。
**根拠種別:** 既存コードパターン（言語ハンドラ構造）+ guardrail 原則（曖昧さ回避）
**選択と理由:** [1] 推奨どおり + コメント本文を `config.docs.defaultLanguage` に揃える（ドキュメント出力言語と一致）。

### Q4. guardrail の配置と強度

**推奨:** 既存 `comment-intent-not-logic` を MUST 表現で拡張する。
**根拠種別:** 既存コードパターン（guardrail 記事構造）
**選択と理由:** [1] 既存記事拡張。関心事が同一のため新規記事追加より統合が適切。MUST 表現で AI に強制する。

### Q5. テスト戦略

**推奨:** 各言語ハンドラの minify に対するユニットテストを追加する。
**根拠種別:** 既存コードパターン（ユニットテスト配置慣習）
**選択と理由:** 推奨どおり。プリフィックス保持・非プリフィックス削除・末尾コメント削除の 3 観点を各言語で検証。

### Q6. 代替案と将来の拡張性

**検討した代替:**
- 最小 2 種 (`WHY:` / `SECURITY:`) → 却下。`HACK:` は既知制約のドキュメント価値が高い。
- 5 種 (Issue 原案) → 却下。`TODO:` は開発者向け、`NOTE:` は乱用リスク。
- 末尾コメント対応 → 却下。AI に「独立行で理由を書く」習慣を促すため。

**将来の拡張:** プリフィックスは後から追加容易（段階的改善のため既存コード移行コスト無し）。運用実績を見て `INVARIANT:` / `PERF:` / `COMPAT:` / `DEPRECATED:` 等の追加を検討。
