# Draft: DataSource ロード失敗のサイレント無視を修正

**開発種別:** バグ修正

**目的:** DataSource モジュール読み込み時のロード失敗（構文エラー等）が通常実行で通知されない問題を解消する。ロード失敗は即座にエラーとして顕在化させ、ユーザーが原因に気付ける状態にする。

## 背景

- DataSource ローダーはモジュール import 時の例外を verbose ログのみで記録し、処理を継続している。
- verbose ログは通常実行では出力されないため、ユーザーは DataSource のロード失敗に気付けない。
- 結果、壊れた DataSource が黙って除外され、サブカテゴリ 0 件などの不明瞭な下流症状を招く。
- 本プロジェクトの方針（CLAUDE.md）に以下の点で反している:
  - 「フォールバック値の抑制」: 必須要素が欠けた状態で黙って動作継続しない
  - 「過剰な防御コードを書かない、内部インターフェースは信頼する」
  - DataSource のロードは内部コード（sdd-forge 自身またはプリセット）であり、境界バリデーション対象ではない

## スコープ

### 対象

- DataSource ローダーにおけるロード失敗時の方針変更（握りつぶし → 呼び出し元への伝播）
- ロード失敗時のユーザー可視性の改善

### 対象外

- ロード失敗時の整形されたエラー表示層（Node 標準のスタックトレースで十分な要件とする）
- opt-in 方式（`--strict` 等）の導入
- サブカテゴリ 0 エントリ時の警告ロジック
- DataSource 以外のモジュールロード箇所の調査

## 要件

優先度は P1（必須）/ P2（推奨）で示す。

- **[P1] When** DataSource モジュールのロードが失敗したとき、**shall** sdd-forge は当該失敗を黙って無視せず、エラーとして顕在化させること。
- **[P1] When** ロード失敗が発生したとき、**shall** ユーザーが問題のあるファイルを特定できる情報（パス相当の識別子）が通常実行のエラー出力に含まれていること。
- **[P1] When** 全 DataSource が正常にロードできる通常ケースにおいて、**shall** 既存の挙動と出力に差異がないこと。
- **[P2] If** 本修正以降にロード失敗が再発した場合、**shall** 既存の自動テスト群（`npm test`）でリグレッションを検出できる状態であること。

## 受け入れ条件

- DataSource モジュールのロード失敗時に、sdd-forge プロセスが失敗として終了し、ユーザーは標準出力/標準エラーで原因を確認できる。
- 正常な DataSource のみで構成されるプロジェクトで `sdd-forge docs build` および `sdd-forge docs scan` が現行と同じ成功結果を返す。
- 全プリセットテスト（`npm test`）がパスする。

## 既存機能への影響

- 正常系（全 DataSource がロード可能）: 影響なし。出力・副作用は現行と同一。
- 異常系（壊れた DataSource が存在）: これまで黙って除外されていたロード失敗が、修正後はエラーとして早期に顕在化する（意図した改善）。
- 影響範囲: DataSource ローダーを経由するコマンド群（docs 系）。成功ケースには影響しない。

## 代替案の評価（意思決定段階）

本ドラフトは実装直前の意思決定段階で整理しており、以下はブレスト中のアイデアではなく採否の結論を示す。

- **ログレベルを上げる案**: 可視性は改善するが「失敗しても処理継続」という本質は残り、下流の 0 件症状も解消しない。**不採用**。
- **`--strict` opt-in 方式**: 安全側がデフォルトでないのは sdd-forge ポリシーに反する。CLI 表面も増える。**不採用**。
- **ローダーでの握りつぶしをやめ例外を伝播させる方針**: プロジェクト方針（フォールバック抑制、過剰防御回避）に合致し、要件充足に最短で到達する。**採用**。

## 将来拡張性

- 「壊れたプリセットを検出しつつ他は動かしたい」需要が将来生じた場合も、ローダー側に分岐を持たせるのではなく、プリセット検証専用コマンド（例: 仮称 `presets lint`）として切り出す方針とする。ランタイム挙動はあくまで fail-fast を維持する。
- CLI トップレベルで整形エラー表示層が必要になった場合は、既存のトップレベル例外処理を拡張して対応する余地がある（本 spec のスコープ外）。

## Q&A

**Q1: 呼び出し元側で追加のエラーハンドリングを加える必要はあるか？**
A: 不要。根拠は (3) 既存コードパターン: 現行の呼び出し元もロード失敗を想定した追加処理を持たず、例外は CLI トップレベルまで自然に伝播する構造になっている。本修正後もこの構造を維持すれば要件を満たす。

**Q2: エラーメッセージに問題ファイルの情報が含まれることはどう担保されるか？**
A: 根拠は (2) プロジェクト方針「過剰な防御コードを書かない」: Node.js が `import()` 失敗時のスタックトレースに該当ファイル情報を含めるため、sdd-forge 側で追加整形を加えなくても要件を満たせる。独自整形はスコープ外。

**Q3: テスト観点はどうあるべきか？**
A: 根拠は (1) docs の SDD ルール「実装検証テストが原則必須」: ロード失敗を起こす状況を再現する自動テストを追加し、リグレッションを継続的に検出可能な状態にする。テストの配置・実装詳細は test フェーズで決定する。

**Q4: alpha 版ポリシー（後方互換コードを書かない）と整合するか？**
A: 根拠は (2) プロジェクト方針の alpha 期間ポリシー: 後方互換性を意図的に下げる変更は alpha 期間に許容される。「過剰な防御コードを書かない」方針とも整合する。
