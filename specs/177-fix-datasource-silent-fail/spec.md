# Feature Specification: 177-fix-datasource-silent-fail

**Feature Branch**: `feature/177-fix-datasource-silent-fail`
**Created**: 2026-04-14
**Status**: Draft
**Input**: GitHub Issue #148 — DataSource ロード失敗がサイレントに無視される

## Goal

DataSource モジュールのロード失敗が通常実行で通知されない問題を解消する。ロード失敗を即座にエラーとして顕在化させ、ユーザーが原因を特定・修正できる状態にする。これにより sdd-forge の「フォールバック値の抑制」「過剰な防御コードを書かない」方針を DataSource ロード経路で一貫させる。

## Scope

- DataSource ローダーにおけるロード失敗時の方針変更（握りつぶし → 呼び出し元への伝播）。
- ロード失敗時のユーザー可視性の改善（標準のプロセス終了とエラー出力による通知）。
- 本変更の回帰を検出するための自動テスト追加。

## Out of Scope

- ロード失敗時の整形されたエラー表示層（Node 標準のスタックトレースで十分）。
- `--strict` 等の opt-in 方式の導入。
- サブカテゴリ 0 エントリ時の警告ロジック。
- DataSource 以外のモジュールロード経路に対する同種の調査・修正。
- `fs.readdirSync` の同期 I/O 使用の置換（pre-existing で本 spec の diff では触っていない）。
- DataSource ファイル数・サイズの上限導入（pre-existing で本 spec の diff では触っていない。プリセット設計上 `data/` 配下のファイル数は有限かつ人手管理される前提）。

## Clarifications (Q&A)

- Q: 呼び出し元側で追加のエラーハンドリングを加える必要があるか？
  - A: 不要。既存コードパターンは例外を CLI トップレベルまで伝播させる構造で、本修正後もそれを維持すれば要件を満たす。
- Q: エラーメッセージに問題ファイルの情報が含まれることはどう担保されるか？
  - A: Node.js が `import()` 失敗時にスタックトレースで該当ファイルを示すため、sdd-forge 側で追加整形は不要。
- Q: alpha 版ポリシーと整合するか？
  - A: 整合する。後方互換性を意図的に下げる変更は alpha 期間に許容される。

## Alternatives Considered

- **ログレベルを上げる案（verbose → warn/error）**: 可視性は改善するが「失敗しても処理継続」という本質的な問題が残り、下流の 0 件症状も解消しない。不採用。
- **`--strict` オプションによる opt-in 方式**: 安全側がデフォルトでないのは sdd-forge ポリシーに反し、CLI 表面も増える。不採用。
- **ローダーでの握りつぶしをやめ例外を伝播させる方針（採用案）**: プロジェクト方針（フォールバック抑制、過剰防御回避、内部インターフェースを信頼する）に合致し、最小変更で要件を満たす。

### Why This Approach

採用案は、sdd-forge の既存ポリシー（CLAUDE.md 記載の「フォールバック値の抑制」「過剰な防御コードを書かない」「内部インターフェースは信頼する」）と完全に一致しており、ローダー経路だけ例外的に握りつぶしていた状態を是正するものである。追加の CLI 表面を導入せず、Node.js 標準のスタックトレースにより必要情報が提供されるため、実装・運用コストが最小となる。

## Requirements

優先度は P1（必須）/ P2（推奨）で示す。

1. **[P1] When** DataSource モジュールのロードが失敗したとき、**shall** sdd-forge は当該失敗を黙って無視せず、エラーとして顕在化させること。
2. **[P1] When** ロード失敗が発生したとき、**shall** ユーザーが問題のあるファイルを特定できる情報（パス相当の識別子）が通常実行のエラー出力に含まれていること。
3. **[P1] When** 全 DataSource が正常にロードできる通常ケースにおいて、**shall** 既存の挙動と出力に差異がないこと。
4. **[P1] When** ロード失敗が発生したとき、**shall** プロセスは非ゼロの終了コードで終了すること（Exit Code Contract）。
5. **[P2] If** 本修正以降にロード失敗が再発した場合、**shall** 既存の自動テスト群（`npm test`）でリグレッションを検出できる状態であること。

## Test Strategy

- **ユニットテスト**: DataSource ローダー公開 API に対し、構文エラーを含むモジュールファイルを配置した一時ディレクトリを入力として与え、例外が送出されることを検証する。正常なモジュールのみを配置したケースでは現行どおり全件ロードされることも確認する。
- **既存テスト**: プリセット整合性テストおよびその他の既存 `npm test` がすべてパスすることを確認する。
- **配置方針**: 「公開 API のコントラクトテスト」に該当するため `tests/` 配下（`npm test` 対象）に配置する。`specs/<spec>/tests/` は採用しない（将来改修でも壊れていれば常にバグ）。

## Acceptance Criteria

- DataSource ローダーに、モジュールロード失敗を握りつぶす処理が存在しない。
- 壊れた DataSource ファイルを含むディレクトリに対しローダーを呼ぶと、例外が呼び出し元に伝播し、sdd-forge プロセスは非ゼロ終了コードで終了する。
- 正常な DataSource のみで構成されるプロジェクトで `sdd-forge docs build` および `sdd-forge docs scan` が現行と同じ成功結果を返す。
- 追加したユニットテストおよび既存の `npm test` がすべてパスする。

## Impact on Existing Features

- 正常系（全 DataSource がロード可能）: 影響なし。出力・副作用は現行と同一。
- 異常系（壊れた DataSource が存在）: これまで黙って除外されていたロード失敗がエラーとして顕在化する。影響を受けるユーザーは当該 DataSource を修正するか除去する必要がある（意図した改善）。
- 影響範囲: DataSource ローダーを経由するコマンド群（`docs build`, `docs scan`, `docs data` 等）。成功ケースには影響しない。

## Migration Plan

本 spec は CLI コマンド・オプションの削除や意味変更を伴わないため、CLI 表面に対する移行手順は不要。
