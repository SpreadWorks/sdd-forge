# Feature Specification: 186-logger-container-service

**Feature Branch**: `feature/186-logger-container-service`
**Created**: 2026-04-17
**Status**: Draft
**Input**: Issue #165 (親タスク #1877 Phase2C)

## Goal

Logger の singleton 実装を廃止し、Container 管理下の通常サービスとして扱う。同時に、状態の二重保持の解消、メトリクス集計の責務分離、ログ出力先のキャッシュ化、呼び出し元スタックフレーム判定のパス表現非依存化を行う。

## Scope

- Logger を Container が所有するインスタンスとして 1 回生成・登録し、以降はすべて Container から取得する経路に統一する。
- Logger 内部で保持している作業ディレクトリ・設定オブジェクト等の、Container と重複する状態を除去する。
- メトリクス集計（AI エージェント呼び出し結果の集計）を Logger からフロー計測の責務を持つレイヤへ移動する。
- Container 初期化時にログ出力先を確定し、以降のログ書き込みで再計算を発生させない。
- 呼び出し元ファイル判定を、スタックフレームのパス表現差（URL 形式、OS 区切り、相対/絶対等）に依存しない同一性判定に切り替える。
- 既存の Logger 呼び出し箇所を Container 経由の取得に統一する。
- 上記変更に対応する単体テストの更新を行う。

## Out of Scope

- Container の取得 API 形式（現行の取得方式）の変更。
- Logger 以外の Container 登録サービスの設計変更。
- ログ出力フォーマット、保存ファイル構成、保存先ポリシーの変更。
- ログ有効フラグの意味論変更。
- 親タスク #1877 の他 Phase。
- ログ内容のマスキング・機密扱い・フィルタリング（観測性・プライバシーの専門領域として別 spec で扱う）。
- 既存テストの意図変更に伴うレビュー記録（本 spec の R7 において、Logger singleton 廃止の結果として既存 Logger 関連テストの書き換えが必要であることはユーザー承認済み（draft フェーズ Q3））。

## Clarifications (Q&A)

- Q1: Issue 記載の 5 タスク（singleton 廃止、状態二重保持解消、メトリクス分離、ログ出力先キャッシュ、呼び出し元判定修正）を同一 spec で扱うか。
  - A: はい。すべて Logger のサービス化という単一の関心に収束しており一体で扱う。
- Q2: Container からの取得はプロパティ形式か既存の取得 API 形式か。
  - A: 既存形式を維持。プロパティアクセス化は全 Container 利用箇所に波及するため別 spec で扱う。
- Q3: ログ機能が無効時のメトリクス集計の扱い。
  - A: メトリクスはログ有効フラグから独立する。ログが無効でもメトリクスは集計する。

## Alternatives Considered

- A. Container 取得 API をプロパティアクセスに変更する案: 不採用。影響範囲が Container 利用箇所全体に及び、本 spec の単一責任を破る。プロパティ化は別 spec として切り出す。
- B. singleton を維持しメトリクス分離のみ行う案: 不採用。状態の二重保持問題が解消されない。
- C. Container 管理 + 既存取得 API 維持（採用）: 既存パターン・コードベースの統一と整合し、波及を最小化しつつ Issue の全タスクを満たす。
- D. 静的関数 API へ置き換える案: 不採用。状態がモジュールスコープのグローバル変数に移るのみで、singleton と同等の結合が残る。

## Requirements

各要件に優先度（P1: 必須 / P2: 関連改善 / P3: 任意改善）を付す。

### R1 (P1): Logger の singleton 廃止

- When Container が初期化される, Container は Logger インスタンスを 1 回だけ生成し自らの管理下に登録する shall
- When コード内の任意の呼び出し元が Logger 機能を利用する, その取得経路は Container からの取得に限定される shall
- When 変更完了後のコードベースを検査する, singleton 取得用の静的 API は存在しない shall

### R2 (P1): 状態の二重保持の解消

- When Logger がログ書き込みに必要なコンテキスト（作業ディレクトリ、ログ有効フラグ、エントリコマンド、フロー参照）を要する, Container 側で解決済みの値のみを参照する shall
- When 変更完了後の Logger を検査する, Container と同一意味の値を Logger 内部で保持していない shall

### R3 (P1): メトリクス集計の責務分離

- When AI エージェント呼び出しが完了する, メトリクス集計はフロー計測の責務を持つレイヤから実行される shall
- When Logger がログ書き込みを行う, Logger はフロー計測状態を更新しない shall
- When ログ機能が無効の状態で AI エージェント呼び出しが完了する, メトリクス集計は実行される shall

### R4 (P2): ログ出力先パスのキャッシュ化

- When Container 初期化が完了する, ログ出力先の値は確定状態で保持される shall
- When Logger が個々のログ書き込みを行う, ログ出力先の再計算は発生しない shall

### R5 (P2): 呼び出し元スタックフレーム判定の同一性判定化

- When Logger が呼び出し元ファイルを特定する, 呼び出し元が Logger 自身のファイルに該当する場合は除外される shall
- When スタックフレームのパス表現が URL 形式・OS 区切り・相対/絶対等の複数形式で現れる, 除外判定はそれら表現の差に依存しない同一性判定に基づく shall

### R6 (P1): 呼び出し箇所の Container 経由への統一

- When 既存モジュールが Logger を参照する, 参照方法は Container 経由の取得に統一される shall
- When 本体コードから Container 外で Logger を生成する経路を検査する, そのような経路は存在しない shall（テストでのインスタンス直接生成は除外）

### R7 (P1): テストの新 API への移行

- When 既存の Logger 関連テストが実行される, テストは singleton API に依存せず新しい提供経路または直接生成を通じて Logger を取得する shall
- When 変更後のテストスイート全体が実行される, 全テストはパスする shall

## Acceptance Criteria

- AC1: Container 初期化を 1 回実行した後、Container から取得した Logger インスタンスは毎回同一参照である。
- AC2: singleton 取得用の静的 API はコードベースから削除されており、参照箇所も存在しない。
- AC3: Logger 生成時に渡された値以外の作業ディレクトリや設定オブジェクトを Logger 内部で保持していない（フィールド検査による確認）。
- AC4: AI エージェント呼び出しを実行したとき、メトリクス集計の呼び出しは Logger 経路ではなくフロー計測側の経路から行われる。
- AC5: ログ機能を無効化した状態で AI エージェント呼び出しを実行したとき、メトリクス集計が実行される。
- AC6: 単一の CLI 実行中に複数回ログが書き込まれても、ログ出力先の解決処理は Container 初期化時の 1 回のみ行われる。
- AC7: Logger 内からのログメソッド呼び出しにおいて、呼び出し元ファイル判定は Logger 自身のファイルを除外し、直近呼び出し元のフレームを返す。スタックフレームのパスが URL 形式と通常パス形式のいずれで現れても結果は同一である。
- AC8: 既存の Logger 呼び出し箇所（docs 系、flow 系、git 系）が Container 経由の取得に置き換わっており、本体コードに singleton API 参照は残っていない。
- AC9: 変更後に `npm test` を実行したとき、全テストがパスする。

## Why This Approach

- 既存の Container 設計は、CLI-wide に初期化されるサービスを 1 箇所で生成し共有する責務を持っており、Logger をその配下に収めることは既存方針と整合する。
- `container.get("name")` 形式は本プロジェクトで既に広範に採用されている統一パターンであり、Logger のみ特別扱いしていた現状の不整合を解消する。
- 状態の二重保持はバグの温床であり、Container 側に真実を 1 つ置くことで、将来の設定反映経路の変更時に更新漏れが起きにくくなる。
- メトリクス集計は本来 AI エージェント呼び出し結果をフロー計測に反映する処理であり、ログ出力（観測性）とは責務が異なる。Logger は書き込みのみに専念させることで単一責任を強化する。
- ログ出力先のキャッシュ化は、頻繁なログ書き込み時の不要な再計算（worktree 判定を含む）を取り除き、I/O 以外のコストを最小化する。
- 呼び出し元判定の同一性判定化は、将来の Node.js バージョン・OS・実行形態差での挙動揺らぎを防ぐ。

## Impact on Existing Features

- Logger を singleton 経由で直接参照していたコード（docs 系ディスパッチャ、flow step 記録、git 補助）はすべて Container 経由の取得に置き換える必要がある。alpha 版ポリシーにより旧 API の互換ラッパーは残さない。
- Logger 関連の単体テストは新 API に書き換える必要がある。
- ログ機能が無効の状態での AI エージェント呼び出しでは、従来スキップされていたメトリクス集計が今後は実行される。これは意図的な挙動変更であり、Requirements に明記する。
- ログ出力先の動的変更（ランタイム中に設定を書き換えて即反映）を前提とするユースケースは本変更後サポートされない。現状コードベースに該当ユースケースは存在しない。

## Test Strategy

- Logger のエージェント書き込み、Git 書き込み、イベント書き込みの各経路について、期待されるログ行・成果物が生成されることを単体テストで検証する。
- ログ有効フラグが無効な状態で各ログメソッドを呼んだとき、I/O 副作用が発生しないことを検証する。
- 呼び出し元ファイル判定について、スタックフレームのパスが URL 形式の場合と通常パス形式の場合の双方で、Logger 自身のフレームが除外されることを検証する。
- AI エージェント呼び出し経路について、メトリクス集計呼び出しが Logger 経由ではなくフロー計測側で発生することをテストダブルで検証する。ログ無効状態でもメトリクス集計が発生することを確認する。
- Container 初期化からの Logger 取得が毎回同一参照であることを検証する。
- 変更後のテストスイート全体（`npm test`）が回帰なくパスすることを確認する。
