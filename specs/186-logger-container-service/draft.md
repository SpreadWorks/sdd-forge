# Draft: Logger を Container 配下のサービスに移行

**開発種別:** enhancement (refactor)

**目的:** Logger の singleton 実装を廃止し、Container 管理下の通常サービスとして扱う。併せて責務分離・状態の二重保持解消・初期化コストの削減・呼び出し元判定の同一性判定化を行う。

## 背景

親タスク #1877 Phase2C (Issue #165)。現状の Logger は singleton として実装され、Container にも登録されているが呼び出し側は singleton を直接参照している。Container が保持する設定情報と Logger 内部の状態が重複し、ログ機能にフロー計測機能が混在している。初期化済み値のキャッシュ化、および呼び出し元フレームの判定方式にも改善余地がある。

## スコープ

### 対象

本 spec が扱う要件は Logger 周辺の以下の関心事に限る:

1. Logger の singleton 廃止と Container 経由の提供
2. Logger と Container 間の状態重複の解消
3. Logger から FlowManager メトリクス集計への依存の分離
4. ログ出力先パスの初期化時確定（キャッシュ化）
5. 呼び出し元スタックフレーム判定の同一性判定化
6. 既存の Logger 呼び出し箇所の Container 経由への置き換え
7. 上記変更に伴う単体テストの更新

### スコープ外

- Container の取得 API 形式変更（既存の取得 API を維持）
- Logger 以外の Container 登録サービスの設計変更
- ログ出力フォーマット・保存ファイル構成・保存先ポリシーの変更
- `cfg.logs.enabled` の意味論変更
- 親タスク #1877 の他 Phase

## 要件

各要件は優先度付き。P1 は本 spec の必須要件、P2 は関連改善、P3 は任意改善。

### R1 (P1): Logger の singleton 廃止
- When Container の初期化が実行される, Container は Logger インスタンスを1回だけ生成し管理下に登録する shall
- When 任意のコードが Logger を利用する, Container から取得した共通インスタンスを使用する shall
- When コードベースに singleton 取得 API が存在する, 本変更後に当該 API は存在しない shall

### R2 (P1): 状態の二重保持の解消
- When Logger がログ書き込みに必要なコンテキスト（作業ディレクトリ、ログ有効フラグ、エントリコマンド、フロー参照）を要する, Container 側で解決済みの値を Logger 生成時に受け取る shall
- When Logger が内部で作業ディレクトリや設定オブジェクトを保持している場合, 本変更後に Logger は Container と重複する状態を保持しない shall

### R3 (P1): メトリクス集計の責務分離
- When AI エージェント呼び出しが完了する, メトリクス集計は Logger ではなく Agent 呼び出しの責務として FlowManager に対して実行される shall
- When Logger がログ書き込みを行う, Logger はフロー計測状態を更新しない shall
- When ログ機能が無効化されている, メトリクス集計はログ有効フラグと独立に実行される shall

### R4 (P2): ログ出力先パスのキャッシュ化
- When Container 初期化が完了する, ログ出力先の値は確定状態で保持される shall
- When Logger が個々のログ書き込みを行う, ログ出力先の再計算は発生しない shall
- Rationale: 現状は毎ログ書き込み時に冗長な解決処理を経由している

### R5 (P2): 呼び出し元スタックフレーム判定の同一性判定化
- When Logger が呼び出し元ファイルを特定する, 呼び出し元が Logger 自身のファイルに該当する場合は除外される shall
- When スタックフレームのパス表現が URL 形式・OS 区切り・相対/絶対等の複数形式で現れる, 除外判定はそれら表現の差に依存しない同一性判定に基づく shall

### R6 (P1): 呼び出し箇所の Container 経由への統一
- When 既存モジュールが Logger を参照する, 参照方法は Container 経由の取得に統一される shall
- When Container 外で Logger を生成する経路が存在する, 本変更後に当該経路は存在しない shall（テストを除く）

### R7 (P1): テストの新 API への移行
- When 既存の Logger 関連テストが実行される, テストは singleton API に依存せず新しい提供経路を通じて Logger を取得・生成する shall
- When 変更後のテストスイート全体が実行される, 全テストはパスする shall

## 既存機能への影響

- When 既存コードが singleton 取得 API を直呼びしている, 本変更後にそれらの呼び出しはコンパイル（実行）時エラーとなる。該当箇所はすべて Container 経由に置き換える。
- When Logger 関連のテストが既存コードで singleton を前提としている, テストコードを新 API に書き換える。
- When ログ出力先設定を CLI 実行中に動的変更するユースケースが想定されていた場合, 変更は反映されない（初期化時点の値を使用）。現状のコードベースにこのユースケースは存在しない。
- When ログ機能が無効の状態で AI エージェントが呼ばれる, メトリクス集計は従来スキップされていたが本変更後は実行される。意図的な挙動変更であり、ログの有無とメトリクスは独立とする方針に従う。

## 制約

- alpha 版ポリシーにより、後方互換のための旧 API ラッパーは残さない。
- 外部依存を追加しない（Node.js 組み込みのみ）。
- OOP による型表現の方針に従い、Logger はクラスとして維持する。
- Logger の公開機能（エージェントログ、Git ログ、イベントログ、書き込み完了待機、初期化状態取得）の役割は維持する。

## エッジケース

- When Container 初期化時に設定ファイルが未ロード, Logger はログ無効状態で生成され、メソッドは no-op となる shall
- When worktree 内で CLI が実行される, ログ出力先は worktree ではなく主リポジトリ側に決定される shall
- When 書き込み待ちの I/O が残っている, 書き込み完了待機機能は全ての未完了書き込みを待つ shall
- When AI エージェント呼び出しが例外で失敗する, メトリクス集計はスキップされる shall

## テスト戦略

- Logger の各ログ経路（エージェント、Git、イベント）について、新 API 配下で期待される出力が得られることを検証する。
- ログ機能が無効な状態でメソッドが副作用を発生させないことを検証する。
- 呼び出し元ファイル判定が Logger 自身のフレームを除外することを検証する。
- メトリクス集計が Agent 呼び出し経路で実行されることを検証する。
- 全体回帰が発生しないことを既存テストスイートの実行により確認する。

## 代替案の検討

| 案 | 評価 | 根拠 |
|---|---|---|
| A. Container 取得 API をプロパティアクセスに変更 | 不採用 | 全 Container 利用箇所に波及するため本 spec のスコープを超える。別 spec 候補。|
| B. singleton を維持しメトリクス分離のみ行う | 不採用 | 状態の二重保持が残り本 Issue の要求 1, 2 を満たさない。|
| **C (採用). Container 管理 + 既存取得 API 維持** | 採用 | 既存パターンと整合し波及を最小化。Issue の 5 タスクすべてを満たす。|
| D. 静的関数 API へ置き換え | 不採用 | 状態がモジュールスコープのグローバル変数に移るのみで、singleton と同等の問題が残る。|

## 将来の拡張性

- When Container の取得 API がプロパティアクセス化される別 spec が立つ, Logger は既に通常サービス化されているため他サービスと同じ移行経路で対応できる。
- When メトリクス集計の対象や経路（非エージェント経由の集計等）が拡張される, FlowManager 側に責務が集約されているため追加先が一意となる。
- When Logger のテストダブル注入が必要な将来要件が発生する, 依存値が生成時に明示される構造のためテスト差し替えが容易となる。
