# Draft: レポートの Tests 欄にテスト結果を記録する

**開発種別:** Enhancement（スキルテンプレート修正）

**目的:** flow-plan テストフェーズ完了時に test-summary を記録する指示をスキルに追加し、レポートの Tests 欄が常に `-` になる問題を解消する。

## 背景

レポートの Tests 欄が常に `-` になっている。原因はテストフェーズで test-summary が記録されないこと。テンプレートには既に記録指示が存在するが、インストール済みスキルに同指示が欠落している。

## 要件（優先順）

1. **【必須】** flow-plan テストフェーズ完了時（テスト仕様書 README を作成した後）に、書いたテスト数を unit/integration/acceptance 別に記録しなければならない
2. **【必須】** テスト数を記録する際は（When: テストフェーズでテスト数を記録するとき）、テストランナー・言語固有の出力（TAP 等）に依存する手段を使ってはならない（shall: テスト仕様書執筆時に把握できる情報のみで記録する）
3. **【任意】** flow-impl フェーズでの記録更新は行わない

## Q&A

**Q1: test-summary に記録する数値は「書いたテスト数」か「実行結果（pass/fail）」か？**
A: 書いたテスト数。根拠: ガードレール原則「Single Responsibility」より、sdd-forge は特定プロジェクトの実装に依存してはならない。TAP 出力パースはプロジェクト固有のため採用しない。

**Q2: 自動カウントは可能か？**
A: テスト仕様書（README）を書く時点で AI はテスト数を把握しているため、その時点での記録で十分。根拠: 既存の flow-plan テストフェーズで README 作成が既に必須要件であり（既存コードパターン）、カウント情報は自然に得られる。

**Q3: flow-impl にも test-summary 更新指示を追加するか？**
A: しない。plan フェーズの記録で十分。根拠: 追加の記録タイミングは必要性が確認されておらず、スコープを広げることはガードレール「Single Responsibility」に反する。

## 影響範囲

- スキルテンプレート（flow-plan）: 変更なし（既に正しい）
- インストール済みスキルの同期: 必要
- 既存機能への影響: なし
