# Draft: auto mode — テスト中に発見したバグの自動記録

## 背景

Issue #54: spec 111 の auto mode でテストを書いた際、プロダクトコードのバグを2件発見したが、
spec のスコープ外だったためテストを現在動作に合わせて修正した。
バグは redolog にもボードにも記録されず見落とされた。

## 問題

auto mode では人間が見ていないため、テスト失敗→テスト修正で終わり、発見した問題が消える。
これはテスト中だけでなく、worktree 操作やマージ、コミットなどフロー全体で起きうる問題。

## 要件

### ゴール
フロー実行中にエラーや問題を発見した場合、redolog に記録するトリガー条件を明確化し、
auto mode でも問題が見落とされないようにする。

### スコープ
- `src/templates/partials/redo-recording.md` を拡充し、具体的なトリガー条件一覧を追加
- 各スキル（flow-plan, flow-impl, flow-finalize）側には最小限の補足のみ

### スコープ外
- redolog の名前変更（別タスク: ボード 2dbb）
- redolog.json のフォーマット変更
- 新しいコマンドの追加

## 決定事項

1. 記録先は既存の redolog（`sdd-forge flow set redo`）を使う。新しいファイルやコマンドは追加しない
2. `redo-recording.md` パーシャルにトリガー条件を集約し、各スキル側の変更は最小限にする
3. 対象スキルは flow-plan, flow-impl, flow-finalize の3つ
4. redolog の名前の違和感は別タスク（ボード 2dbb）で対応する

## 追加するトリガー条件（案）

### テスト関連
- テスト失敗の調査でプロダクトコードのバグを発見したが、現在の spec スコープ外で修正できない場合
- テストを現在の動作に合わせて調整した場合（本来の期待動作と異なる場合）

### コマンド・操作関連
- worktree の作成・削除でエラーが発生した場合
- マージでコンフリクトが発生した場合
- コミットが失敗した場合（hook 失敗含む）
- gate が FAIL した場合（既存）

### 共通
- 設計上の前提が誤っていたことが判明した場合（既存）
- 回避策（workaround）を適用した場合

## 既存機能への影響

- `redo-recording.md` パーシャルの内容拡充のみ。コマンドやフォーマットの変更なし
- `sdd-forge upgrade` 実行後にユーザーのスキルファイルに反映される
- 既存の redolog.json との互換性に影響なし

## テスト戦略

- SKILL.md はテンプレートであり、AI への指示文書。ユニットテストの対象ではない
- `sdd-forge upgrade` で正しく反映されることの確認（既存の upgrade テストでカバー）
- パーシャルの include が正しく展開されることの確認
