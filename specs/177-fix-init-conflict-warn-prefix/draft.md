# Draft: init conflictsExist メッセージの ERROR プレフィックスを WARN に変更

**開発種別:** バグ修正

**目的:** `sdd-forge docs init` が既存ファイルと conflict した際に出す通知メッセージが「エラー」として分類されないようにし、CI 等の汎用的なログスキャンで誤検知されないようにする。

## 議論モード

**本ドラフトは「最終決定」モードで記述されている。** 以下の Q&A および代替案評価はブレインストーミングではなく、ユーザー承認済みの決定事項として扱う。

## 背景

Issue #149: init コマンドの conflict 検出時通知は「エラー」として表示されるが、実態は「スキップ通知」でありプロセスは正常終了する。外部プロジェクトから「CI で `grep error` すると誤検知する」との指摘があった。

sdd-forge 内には「既に存在する系の通知は warning レベルで出す」既存慣習が存在する。init の conflict 通知はこの慣習から逸脱している局所的な不整合である。

## スコープ

### In Scope
- conflict 検出時の通知メッセージ（ja / en 両方）のプレフィックスを、既存慣習に整合する「警告」レベルに変更する

### Out of Scope
- 実際にプロセスを停止する致命的エラーメッセージ（`noType`, `noTemplates` 等）のプレフィックス見直し
- ログ出力ヘルパーの再設計
- 構造化ログレベル API の導入

## Requirements

- **R1 (P1 / Must):** When `sdd-forge docs init` が既存ファイルとの conflict を検出し通知メッセージを出力するとき, the system shall 当該メッセージのプレフィックスとして「エラー」を示す語（`ERROR:`）を使用しない。
- **R2 (P1 / Must):** If conflict 検出時の通知メッセージを出力する, the system shall プロジェクト内の他の「既に存在する」系通知と同じ警告プレフィックス（`WARN:`）を使用する。
- **R3 (P1 / Must):** If 通知メッセージが複数ロケールで提供される, the system shall 全ロケールで同一のプレフィックス表記を使用する。
- **R4 (P1 / Must):** When conflict を検出してもプロセスが正常終了する, the system shall 既存の検出・スキップ・書き込み・終了コードの振る舞いを変更しない。

（本 issue は単一関心事のバグ修正であり、4 件すべてが同一優先度 P1 として扱われる。段階的リリースや部分採用の余地はない。）

## Q&A

### Q1: Goal & Scope — 目的とスコープは明確か？

A. 明確。conflict 通知のプレフィックスのみを既存慣習に整合させる。
- **根拠（Issue #149 本文）:** conflict 通知はプロセスを停止しない単なるスキップ通知であると明記されている。
- **根拠（既存コード）:** conflict 検出ブロック（init の処理）は該当ファイルを `continue` でスキップし、プロセスは正常終了する。

### Q2: Impact on existing — 既存機能への影響は？

A. 通知文言のみの変更であり、動作（検出・スキップ・終了コード）は不変。
- **根拠（Issue #149 本文）:** 現状の処理はスキップしつつ正常終了する旨が明記されている。
- **根拠（原則: alpha 版ポリシー）:** 後方互換コードは書かない方針であり、文言の直接置換が適切。

### Q3: Constraints — 制約は？

A. プロジェクト既存慣習（`WARN:` プレフィックス）との整合性を保つ。全ロケールを同期する。
- **根拠（既存コード）:** flow 初期化時の「preparing flow(s) already exist」通知が `WARN:` プレフィックスを採用している。同種の「既に存在する」通知はこの慣習に従う。
- **根拠（原則: コード品質の維持）:** 「新しいコードは既存のコードパターン・命名規約に合わせる」方針に従い、レビュアー提案の `INFO` / `NOTE` ではなく既存慣習と一致する `WARN` を採用する。

### Q4: Edge cases — エッジケース？

A. 特になし。
- **根拠（Issue #149 本文）:** 対象は単一の通知メッセージで、境界条件や分岐への波及要素は存在しない。

### Q5: Test strategy — テスト戦略？

A. 対象ロケール文言が warning レベルのプレフィックスで始まり、error レベルのプレフィックスを含まないことを保証するテストを追加する。
- **根拠（原則: spec ローカルテスト方針）:** bug 再現テストは「この spec の要件が満たされていることを検証するテスト」であり、長期メンテ対象ではないため spec ローカルに配置する、という SDD テスト配置ルールに従う。

### Q6: Alternatives considered — 代替案？

（以下はいずれも**最終決定として却下**された代替案である。）

1. `INFO:` / `NOTE:` プレフィックス — レビュアー提案。**最終決定: 却下。**
   - **根拠（既存慣習）:** sdd-forge 内の「既に存在する」通知は warning レベル慣習。
2. プレフィックスなし — **最終決定: 却下。**
   - **根拠（既存慣習）:** 他 notification 系が warning プレフィックスを持ち、無プレフィックスは混在を生む。
3. 構造化ログレベル API 導入 — **最終決定: 却下。**
   - **根拠（原則: 本 issue のゴール）:** スコープ外の大規模変更であり、issue の目的（誤検知の解消）を超える。

### Q7: Future extensibility — 将来的な拡張性？

A. warning プレフィックス統一は、将来の構造化ログレベル化の布石となる。
- **根拠（既存慣習）:** sdd-forge ではプレフィックス文字列が事実上のログレベルタグとして機能しており、統一により将来の正規化コストが下がる。
