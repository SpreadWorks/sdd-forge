# Draft: Phase0 小規模クリーンアップ

**開発種別:** リファクタリング（dead code 削除・API 整理・定数集約）

**目的:** 1877 Container 導入に先行し、Container 設計判断とは独立して片付けられる 4 件の小規模修正をまとめて完了し、後続 Phase の差分を見通しやすくする。

## 単一関心（Single Responsibility）の根拠

表面的には 4 修正だが、共通の単一関心は **「1877 Phase 0: Container 導入前に除去すべき dead code / API ノイズの一括クリーンアップ」** である。

- いずれも `.tmp/1877.md` で同一フェーズ（Phase 0）にグルーピングされており、issue #154 も 1 チケットで受領している
- どれも挙動変更を伴わない軽量リファクタで、テスト負荷・レビュー負荷ともに小さい
- 分割すると PR 4 本が発生し、Container 着手までの引き延ばしコストが実作業より大きくなる

この根拠により本 spec は 4 修正を 1 つの関心として扱う。

## 事前調査（Explore Before Asking）

- 終了コード定数の参照範囲をコードベース全体で探索し、影響ファイル数を確認済み
- 集約先となる共通定数モジュールの存在・構造（凍結配列の集約パターン）を確認済み
- リポジトリルート解決関数の実装と呼び出し側の引数使用状況を確認済み
- CLI エントリの help 処理経路が 2 系統に分裂している状態を確認済み
- 削除対象の補助関数群に対する参照がコードベース全体で 0 件（定義のみ・配線ゼロ）であることを確認済み

ユーザーへ確認を残したのは **設計判断として選択肢を分岐させる 1 点** のみ（Q2: 統合時に後方互換層を置くか否か）。他 3 要件は issue #154 本文と `.tmp/1877.md` の既決方針に従い確定可能と判断した。

## 優先順位

| 優先 | 要求 | 根拠 |
|---|---|---|
| P1 | R1: 終了コード定数の単一ソース化 | 影響ファイル数が多く、他要件との import 衝突を避けるため最初に確定させる |
| P1 | R4: 配線ゼロ補助関数の除去 | 他要件と干渉しない独立作業。並行可能 |
| P2 | R2: ルート解決関数の引数整理 | 局所的で影響小 |
| P2 | R3: help 処理経路の単一化 | CLI エントリ 1 ファイル内で完結 |

P1/P2 は実施順の参考であり、全要件を本 PR で完了させる。

## 要件

### R1（P1）: 終了コード定数の単一ソース化

- **When** コードベース内で終了コード定数（成功・失敗）を参照するとき、**shall** 参照先は共通定数モジュール 1 箇所に統一されている。
- **When** 本 spec 完了後、**shall** 旧専用モジュールは削除され、同モジュールを指す import 文はコードベース内に 1 件も存在しない。

### R2（P2）: リポジトリルート解決関数の引数整理

- **When** リポジトリルート解決関数が呼ばれるとき、**shall** 呼び出し側は未使用の引数を渡さない形に揃っている。
- **When** 本 spec 完了後、**shall** 同関数の戻り値は、以下 3 条件で同一結果を返す: (a) 環境変数指定あり、(b) 環境変数なしかつ git リポジトリ内、(c) 環境変数なしかつ git リポジトリ外。

### R3（P2）: help 処理経路の単一化

- **When** help 要求（引数なし、`-h`、`--help`、`help` サブコマンド、`help <topic>`）が与えられたとき、**shall** いずれの入力でも help 出力（内容・終了コード）が同一である。
- **When** 本 spec 完了後、**shall** help 処理は CLI エントリ内の 1 つの分岐ブロックのみに存在する。

### R4（P1）: 配線ゼロの補助関数の除去

- **When** 本 spec 完了後、**shall** コードベース全体で該当補助関数名への参照（定義・import・呼び出し）は 0 件である。
- **If** 将来同機能が必要になった場合、**shall** board `909f` に保存された復活用資料から再実装できる（本 spec スコープ外）。

## 受け入れ基準（Acceptance Criteria）

- `npm test` がパスする
- help 要求の 5 入力パターン（引数なし / `-h` / `--help` / `help` / `help <topic>`）がいずれも同一の help 出力と終了コードを返す
- 未知サブコマンドは失敗終了コードで終了する
- R1・R2・R3・R4 の各「本 spec 完了後」の存在条件が全て満たされていることをコード検索で確認できる

## スコープ外

- Container 設計・導入（Phase 1 以降）
- agent API 呼び出し引数の整理・provider クラス化（Phase 2A）
- `.tmp/1877.md` 指摘 5〜31

## 既存への影響

- R1: 参照 import の一括置換のみ。モジュール解決以外の挙動変化なし
- R2: 呼び出し側から未使用引数が消えるのみ。戻り値セマンティクスは不変
- R3: help 処理経路が 1 箇所に集約。出力内容・終了コードは不変
- R4: 参照ゼロのため挙動影響なし

## 制約

- 外部依存追加なし（Node.js 組み込みのみ）
- alpha ポリシー: 後方互換層を作らない（旧モジュールからの再エクスポート等は不可）
- コミットメッセージは英語

## エッジケース

- help 要求に追加引数（`help <topic>` 形式）が付与されても出力が変化しないこと
- 呼び出し元のファイル位置（symlink 経由 / worktree 内 / パッケージ install 配下）に依らずルート解決が元仕様通り動くこと
