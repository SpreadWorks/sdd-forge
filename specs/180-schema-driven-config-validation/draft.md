# Draft: schema-driven-config-validation

**開発種別:** enhancement
**目的:** config のバリデーション定義が複数箇所に分散している問題を解消し、単一のスキーマ定義を SSOT としたバリデーションに転換する

## 要件サマリー

### 既存機能への影響

- 既知フィールドに対するバリデーションの振る舞いは変わらない
- 未知フィールドが新たに検出・拒否されるようになる（alpha 版ポリシーにより許容）
- バリデーション関数の呼び出しインターフェースが変更される（alpha 版ポリシーにより後方互換不要）

### 要件（優先順）

1. **[Must]** config のバリデーション時に、全フィールドの型・必須・許容値を単一のスキーマ定義に基づいて検証すること
2. **[Must]** config のバリデーション時に、スキーマに定義されていないフィールドが存在する場合、error として throw すること
3. **[Must]** config のバリデーション時に、非推奨フィールドが存在する場合、error として throw すること（warn は設けない）
4. **[Must]** config のバリデーション時に、以下のフィールド間整合性を検証すること: (a) defaultLanguage が languages 配列に含まれること、(b) profiles で参照される provider が providers に定義されていること
5. **[Should]** スキーマ駆動バリデーションを実装する際に、スキーマ定義は公開 API に含めないこと
6. **[Should]** スキーマ駆動化により、コードから参照されなくなったテンプレートファイルおよび手書きバリデーションコードが存在する場合、削除すること

### 制約

- 外部依存なし（Node.js 組み込みモジュールのみ）
- alpha 版ポリシー: 後方互換コードは書かない
- 既存の config フィールド構造は変更しない
- 型定義ファイル内の他のヘルパー関数の整理は本 spec のスコープ外（別タスク board 2268）

### 設計方針（ユーザー決定事項）

- スキーマは JSON Schema 仕様のサブセットに準拠すること
- バリデーション失敗は全て error として扱い、warn レベルは設けないこと
