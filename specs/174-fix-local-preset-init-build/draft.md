# Draft: プロジェクトローカルプリセットの init/build 対応

**開発種別:** バグ修正  
**目的:** `.sdd-forge/presets/<name>/` に配置したプロジェクトローカルプリセットが `sdd-forge docs init` / `docs build` で解決されないバグを修正する。

## 背景

`sdd-forge` はプロジェクト固有のプリセットを `.sdd-forge/presets/<name>/` に配置することでカスタマイズできる設計になっている。

`docs scan` ではこの仕組みが正しく動作しているが、`docs init`（および `docs build` が内部で呼ぶ init 処理）ではプロジェクトローカルプリセットが見つからず `Preset not found` エラーで失敗する。同様の問題が `setup` コマンドにも存在する。

## スコープ

**対象:**
- `docs init` / `docs build` コマンドのプロジェクトローカルプリセット解決
- `setup` コマンドのプロジェクトローカルプリセット対応

**スコープ外:**
- プロジェクトローカルプリセットの UI 選択（setupウィザードでの一覧表示）
- `docs scan` および DataSource 解決（既に正しく動作している）

## 要件（優先順）

**P1（必須 — バグ修正の核心）:**

1. When プロジェクトローカルプリセットを `type` に指定して `sdd-forge docs init` を実行したとき、`Preset not found` エラーが発生せず、`docs/` 配下にチャプターファイルが生成されること
2. When プロジェクトローカルプリセットを `type` に指定して `sdd-forge docs build` を実行したとき、パイプライン全体がエラーなく完了し、ドキュメントが出力されること

**P2（整合性 — 一貫性確保）:**

3. When プロジェクトローカルプリセットを `type` に含む状態で `sdd-forge setup` を実行したとき、設定確認画面および設定ファイルの書き込みが `Preset not found` エラーなく完了すること

**P3（回帰防止）:**

4. If プロジェクトローカルプリセットを使用していない既存プロジェクトで `docs init` / `docs build` / `setup` を実行したとき、今回の変更前と全く同じ結果が得られること

## テスト戦略

- **種別:** ユニットテスト（正式な `npm test` スイートに追加）
- **検証内容:**
  - When プロジェクトローカルプリセットが存在する状態でプロジェクトルートを指定したとき、そのプリセットのテンプレートが解決に使われること
  - When プロジェクトローカルプリセットが存在する状態でプロジェクトルートを指定したとき、そのプリセットの章定義が反映されること
  - If プロジェクトルートを指定しない場合、組み込みプリセットのみを使う従来の動作と同じ結果が得られること

## 既存機能への影響

- プロジェクトローカルプリセットを使用しない既存プロジェクトへの影響なし
- `docs scan` / DataSource 解決など既に正しく動作しているコードパスへの変更なし

## Q&A

**Q1: `setup` コマンドもスコープに含めるか？**  
A（推奨: 含める）— 根拠: 既存コードパターンとして `docs scan` / DataSource 解決がプロジェクトルートを一貫して活用している。`setup` だけを除外すると将来の混乱を招くため、一貫性を優先して含める。

**Q2: テストはどこに置くか？**  
A（推奨: 正式テストスイートへ追加）— 根拠: `src/AGENTS.md` の方針「公開インターフェースの契約テストは正式なテストスイートに置く」に従う。今回の修正は将来の変更でも保たれるべき契約のため、`npm test` で継続的に検証すべき。
