# Draft: 189-unify-command-classes

**Input:** GitHub Issue #168 — spec 188 Phase D フォローアップ (15 コマンドの Command クラス化 + docs build 共通契約化)

**開発種別:** リファクタリング（Phase D 移行）
**目的:** spec 188 が導入した統一実行機構に対し、橋渡しアダプタで接続されている旧スタイルのコマンド群を、統一機構が想定する標準形に全面移行する。併せて一部残存している専用パイプライン経路を共通契約に寄せ、過渡期の橋渡しコードを除去する。これにより spec 188 の「単一コンテキスト取得」要件と受入基準を完全達成する。

## Requirements Priority (高→低)

優先度の高い順に列挙する。本 draft における各要件の優先度は以下に従う。

1. **[最優先] 旧スタイル排除**: When 移行完了後にコマンド実装ファイルを静的検査したとき、shall 旧 main export パターンおよびコマンド内部での共有状態直再解決パターンが 0 件である。
2. **橋渡しコード除去**: When 移行完了後にソースツリー全体を静的検査したとき、shall 過渡期の橋渡しモジュールが削除され、その関数名への import / 参照が 0 件である。
3. **専用パイプライン経路の共通契約化**: When 統一機構のディスパッチャを検査したとき、shall ディスパッチャ内部に特定コマンドに固有の専用経路が存在せず、全コマンドが同一契約で解決される。
4. **外部互換の維持**: When 同一 CLI 引数列で移行前後のコマンドを実行したとき、shall 受理サブコマンド・正常系 stdout・エラー系 stderr・終了コード・ファイルシステム副作用が完全一致する。If 差分が生じる場合、shall spec にその差分とマイグレーション方針を明示列挙する。
5. **検証可能性の担保**: When 合否判定が必要なとき、shall spec 188 の静的検査および本 spec 固有の静的検査によって機械的に判定可能であり、既存テスト全体の結果が baseline から後退しない。
