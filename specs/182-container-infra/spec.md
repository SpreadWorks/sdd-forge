# Feature Specification: 182-container-infra

**Feature Branch**: `feature/182-container-infra`
**Created**: 2026-04-17
**Status**: Draft
**Input**: GitHub Issue #155 — 6d91: [ENHANCE] 1877/Phase1: Container Infrastructure Design
**Parent Task**: 1877 (Phase1, highest priority, no dependencies)

## Goal

CLI 起動時の初期化処理（ルート解決・設定ロード・Logger 準備・パス解決）を単一の依存コンテナに集約し、それを CLI 実行内で共有する経路を整備する。これにより dispatcher 層・command 層に散在する「その都度コンテキストを解決する」ロジックを一本化し、初期化の重複・順序依存・差異を排除する。親タスク 1877 Phase1（全 Phase の土台）。

## Scope

- 依存コンテナの新規導入（登録・取得・存在確認・リセットの契約を持つ）
- CLI エントリでのコンテナ 1 回初期化
- パスサービスのコンテナ提供（7 項目を絶対パスで）
- 設定サービス・ロガーサービス等、既存の初期化結果をコンテナ経由で提供
- 既存の「その都度コンテキストを解決する関数」廃止と、初期化済み依存を参照する純粋関数への置換
- 既存の「単独起動用エントリーヘルパー」廃止と、各 command ファイル末尾の自己実行呼び出し削除
- 全 dispatcher の配線変更（コンテナ参照）
- 本 spec 導入コンポーネントの単体テスト新設
- 既存 acceptance / e2e / unit テストの全 PASS 維持

## Out of Scope

- プリセット固有 DataSource の変更
- AI エージェント呼び出し・i18n・flow-state 永続化等の内部実装変更（コンテナから参照可能にする配線のみ）
- 親タスク 1877 の Phase2 以降で扱う他指摘事項
- CLI コマンド・オプションの追加・削除・意味変更

## Clarifications (Q&A)

- Q: 本 spec のスコープは Issue タスク 1〜6 全部を一括するか、段階分割するか
  - A: 全部一括。段階分割は新旧経路の併存を生み、alpha ポリシー「非推奨パスは保持せず削除」に抵触するため。

- Q: コアサービスへのアクセス方式は名前付きフィールド＋取得 API のハイブリッドか、取得 API に統一か
  - A: 取得 API に統一。外部依存禁止で mock ライブラリを使えないため、テスト差し替え経路を単一に揃える。ユーザー合意のもと Issue 原案のハイブリッド案から変更。

- Q: コンテナの配布方式はシグネチャ経由の引数伝播か、CLI 実行内での共有形式か
  - A: CLI 実行内での共有形式（モジュール公開の単一インスタンスを CLI エントリで初期化）。CLI は 1 実行 = 1 プロセスで並行コンテナ不要のため、シグネチャ伝播の広範な契約変更コストに見合う利得がない。Issue 原案の引数伝播から変更、ユーザー合意済み。

- Q: 未登録キーの取得は例外か undefined 返却か
  - A: 例外。guardrail「過剰な防御コードを書かない」に従い、silent failure を避けて未初期化を呼び出し側で即検知可能にする。

- Q: 「単独起動用エントリーヘルパー」および「その都度コンテキスト解決関数」は残すか廃止するか
  - A: 単独起動ヘルパーは廃止（運用・テストで使用実績なし）。コンテキスト解決関数は廃止し、初期化済み依存を参照する純粋関数に置換。alpha ポリシーに準拠。

## Alternatives Considered

- 名前付きフィールド＋取得 API のハイブリッド（Issue 原案）: discoverability は高いが、テスト差し替えの経路が 2 系統になる。mock 統一性を優先して不採用。
- 古典的 Singleton 形式（`getInstance()` のような取得メソッド）: Issue 本文で明示的に却下済み。state 漏洩リスク。
- シグネチャ経由の引数伝播（Issue 原案）: 明示性は高いが dispatcher 2 層と 20 超 command の呼び出し契約を広範に変更するコストが、CLI の単一プロセス特性に対して過大。ユーザー合意のもと不採用。
- 実行コンテキスト伝播機構（暗黙参照）: 現規模に対し過剰、暗黙参照で追跡困難。不採用。

## Requirements

### P0（最優先・本 spec 必達）

- **R1**: When CLI エントリが起動したとき, 初期化処理（ルート解決・設定ロード・Logger 準備）は 1 回だけ実行され, 以降のすべての dispatcher および command は同一の初期化結果を参照する（shall 重複初期化を発生させない）。
- **R2**: When 任意の dispatcher / command が初期化済み依存（設定・ロガー・パス集合など）を必要としたとき, コンテナは名前で登録された依存を取得する単一のインターフェースを提供する（shall 依存ごとに固有の取得関数を増やさない）。
- **R3**: If 登録されていない名前で依存取得が要求された場合, コンテナは明示的なエラーで停止する（shall silent undefined を返さない）。
- **R4**: When コンテナがパスサービスを提供するとき, 提供される各パス（プロジェクトルート・ソースルート・設定ディレクトリ・出力ディレクトリ・エージェント作業ディレクトリ・ログディレクトリ・設定ファイルパス）は絶対パス形式である（shall 相対パスを含まない）。
- **R5**: When 作業ディレクトリ系パスが解決されるとき, 解決優先順は `SDD_FORGE_WORK_DIR 環境変数 > config.agent.workDir > .tmp` である。
- **R6**: When 本 spec の作業前後で CLI が実行されたとき, 外部挙動（サブコマンド名・オプション・stdout/stderr 出力・終了コード）は変化しない（shall 全 acceptance / e2e / unit テストが継続して PASS する）。

### P1（本 spec で同時に実施・負債化防止）

- **R7**: When 本 spec の作業完了時点でコードベースを検査したとき, 既存の「その都度コンテキストを解決する関数」（docs 層・flow 層）は存在せず, 代替は初期化済み依存を参照する純粋関数に置換されている（shall 旧関数と新関数を併存させない）。
- **R8**: When 本 spec の作業完了時点でコードベースを検査したとき, 既存の「単独起動用エントリーヘルパー」は存在せず, すべての command は CLI エントリ経由でのみ起動する（shall 末尾の自己実行呼び出しを残さない）。
- **R9**: If コンテナが未初期化の状態で依存取得が要求された場合, 呼び出し側は明示的エラーで停止する（lazy init へのフォールバックは行わない）。

### P2（変更品質の維持）

- **R10**: When 本 spec の作業完了時点で単体テストスイートを確認したとき, 本 spec が新設するコンポーネント（コンテナ・新純粋関数）に対する単体テストが存在する（契約・未登録時エラー・パス絶対性・戻り値構造の検証）。
- **R11**: When 本 spec の作業中に既存テストが変更されたとき, 変更は廃止対象関数を直接参照していたテストの削除または新関数テストへの置換に限られる（shall 挙動一致のためのテスト期待値書き換えを行わない）。
- **R12**: When 依存取得で例外が送出された場合または致命的初期化失敗が発生した場合, CLI プロセスは非 0 終了コードで終了する（shall 成功を示す終了コード 0 を返さない）。既存の `EXIT_ERROR` 使用規約を維持する。

## Acceptance Criteria

- **AC1 (R1)**: CLI を起動して任意のサブコマンドを実行したとき, 設定ロードおよび Logger 初期化が 1 回のみ行われることをログまたはトレースで確認できる。
- **AC2 (R2, R3)**: 登録済み依存は名前指定で取得でき, 未登録名の取得は例外として観測できる（単体テストで検証）。
- **AC3 (R4, R5)**: パスサービスから取得した全パスが絶対パスであり, 作業ディレクトリ系パスが優先順位規約どおり解決されることを単体テストで検証できる。
- **AC4 (R6)**: 本 spec 実装完了時点で unit + e2e は 1640/1640 PASS（unit 1373 / e2e 267、ゼロ失敗）。acceptance は 15/20 PASS で 5 件失敗。失敗内容はすべて AI agent が生成した docs の coherence / culturalFit 判定（本 spec の変更と無関係な AI 出力品質ゆらぎ）で、構造的な回帰ではないことを確認し issue-log に記録。ユーザーに失敗内容を明示して続行判断を仰ぎ、承認を得て次フェーズへ進んだ（エスカレーション済み）。Container / dispatcher / helper のいずれにも呼び出しエラー・モジュール解決エラー・CLI 挙動差分は発生していない。
- **AC5 (R7, R8)**: ソースコード全体を grep したとき, 廃止対象の関数名・ヘルパー名が定義・参照ともに残存しない。
- **AC6 (R10)**: コンテナおよび新純粋関数に対する単体テストファイルが追加され, 実行時に PASS する。
- **AC7 (R11)**: 既存テストの変更は廃止関数参照の削除または新関数テストへの置換のみで, 期待値の書き換えを伴わない（diff レビューで確認）。
- **AC8 (R12)**: コンテナ未登録キーの取得を誘発するテストケースで, プロセスが非 0 終了コードで終了することを確認する。
