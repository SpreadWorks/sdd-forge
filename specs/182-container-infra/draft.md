# Draft: Container 基盤設計（1877/Phase1）

**開発種別:** ENHANCE（基盤リファクタリング）

**目的:** 各 dispatcher・command で重複している「ルート解決・設定ロード・Logger 初期化」等の初期化処理を、CLI エントリで 1 回だけ構築する単一の依存コンテナに集約し、それを配布する経路を一本化することで、初期化の重複・差異・順序依存を排除する。

## 要件（優先度順）

### P0（最優先・本 spec 必達）

- **R1**: When CLI エントリが起動したとき, 初期化処理（ルート解決・設定ロード・Logger 準備等）は 1 回だけ実行され, 以降のすべての dispatcher および command は同一の初期化結果を参照する（shall 重複初期化を発生させない）。
- **R2**: When 任意の dispatcher / command が初期化済み依存（設定・ロガー・パス集合など）を必要としたとき, コンテナは名前で登録された依存を取得する単一のインターフェースを提供する（shall 依存ごとに固有の取得関数を増やさない）。
- **R3**: If 登録されていない名前で依存取得が要求された場合, コンテナは明示的なエラーで停止する（shall silent undefined を返さない）。根拠: 未初期化を呼び出し側で検知可能にし、silent failure による後段の不明瞭なエラーを防ぐため。
- **R4**: When コンテナがパスサービスを提供するとき, 提供される各パス（プロジェクトルート・ソースルート・設定ディレクトリ・出力ディレクトリ・エージェント作業ディレクトリ・ログディレクトリ・設定ファイルパス）は絶対パス形式である（shall 相対パスを含まない）。
- **R5**: When 作業ディレクトリ系パスが解決されるとき, 解決優先順は `SDD_FORGE_WORK_DIR 環境変数 > config.agent.workDir > .tmp` である（プロジェクト既存規約の維持）。
- **R6**: When 本 spec の作業前後で CLI が実行されたとき, 外部挙動（サブコマンド名・オプション・stdout/stderr 出力・終了コード）は変化しない（shall 全 acceptance / e2e / unit テストが継続して PASS する）。

### P1（本 spec で同時に実施・負債化を防ぐ）

- **R7**: When 本 spec の作業完了時点でコードベースを検査したとき, 既存の「その都度コンテキストを解決する関数」（docs 層・flow 層それぞれに存在）は存在せず, 代替は初期化済み依存を参照する純粋関数に置換されている（shall 旧関数と新関数を併存させない）。
- **R8**: When 本 spec の作業完了時点でコードベースを検査したとき, 既存の「単独起動用エントリーヘルパー」（各 command ファイル末尾で自己実行する仕組み）は存在せず, すべての command は CLI エントリ経由でのみ起動する（shall 末尾の自己実行呼び出しを残さない）。
- **R9**: If コンテナが未初期化の状態で依存取得が要求された場合, 呼び出し側は明示的エラーで停止する（R3 と整合、lazy init へのフォールバックは行わない）。

### P2（変更品質の維持）

- **R10**: When 本 spec の作業完了時点で単体テストスイートを確認したとき, 本 spec が新設するコンポーネント（コンテナ・新純粋関数）に対する単体テストが存在する（契約・未登録時エラー・パス絶対性・戻り値構造の検証）。
- **R11**: When 本 spec の作業中に既存テストが変更されたとき, 変更は廃止対象関数を直接参照していたテストの削除または新関数テストへの置換に限られる（shall 挙動一致のためのテスト期待値書き換えを行わない）。

## 対象外（Out of Scope）

- プリセット固有 DataSource の変更
- AI エージェント呼び出し・i18n・flow-state 永続化等の内部実装変更（コンテナから参照可能にする配線のみ）
- 親タスク 1877 の他指摘事項（Phase2 以降で扱う）

## 既存機能への影響

- **CLI 外部挙動**: 変化なし（R6）
- **内部 API**: 「その都度コンテキスト解決関数」「単独起動ヘルパー」は削除される。alpha ポリシーにより後方互換は取らない
- **テスト**: 既存 acceptance / e2e / unit 全 PASS 維持。新規 unit test 追加、廃止関数を直接テストしていた箇所は新関数テストに置換

## 制約・非機能要件

- 外部依存禁止（Node.js 組込のみ）
- alpha 版ポリシー: 後方互換コード・非推奨パス保持なし
- `src/` のプロジェクト固有情報禁止
- OOP による型表現（`CLAUDE.md` 記載）: コンテナはクラスとし契約を型で表現する

## エッジケース（確定済み）

- **未登録キー取得**: 例外送出（R3 により確定）
- **未初期化状態での取得**: 例外送出（R9 により確定、lazy init しない）
- **設定ファイル不在時（setup 前・help 実行のみ等）**: コンテナは空の設定サービスを保持して初期化を継続する（現行の「config 不在でもベストエフォートで初期化を続ける」挙動を維持）。設定が必要なコマンドは従来どおり実行時に欠落検知で停止する
- **command ファイルの直接起動**: R8 により CLI エントリ経由のみが正規経路。直接起動は想定外で、コンテナ未初期化エラー（R9）で停止する

## Q&A（決定事項・根拠）

各決定の根拠は (a) プロジェクト CLAUDE.md / AGENTS.md / 既存コード規約、(b) guardrail 原則、(c) Issue 本文のいずれかに基づく。

### Q1: スコープ範囲
**決定**: Issue タスク 1〜6 全部を 1 spec に含める。
**根拠**: (a) 途中状態で新旧の初期化経路が併存すると本 spec の目的（初期化経路の一本化）が達成されず、CLAUDE.md の alpha ポリシー「非推奨パスは保持せず削除」に抵触する。

### Q2: パスサービスのフィールド
**決定**: 7 項目（R4 記載）。
**根拠**: (a) 既存コードの共通パスヘルパーが返す値の集合、Issue 本文の提示項目と一致。

### Q3: コアサービスへのアクセス方式
**決定**: 単一の取得インターフェースに統一。
**根拠**: (a) 外部依存禁止（CLAUDE.md）のため mock ライブラリが使えず、テストでの差し替えを単一経路に揃える必要がある。

### Q4: 取得時のセマンティクス
**決定**: 未登録は例外送出、パスサービスはグループ単位で提供。
**根拠**: (b) guardrail「過剰な防御コードを書かない」、CLAUDE.md「内部インターフェースは信頼する」。

### Q5: コンテナの配布方法
**決定**: CLI 実行内で単一インスタンスが共有される形式。
**根拠**: (c) Issue が古典的 Singleton 形式を却下、(a) CLI は 1 実行 = 1 プロセスのため並行コンテナ不要。ユーザー合意のもと Issue 原案のシグネチャ伝播方式から変更（note 記録済み）。

### Q6: 旧関数・ヘルパーの扱い
**決定**: 単独起動ヘルパー廃止、コンテキスト解決関数は純粋関数に置換。
**根拠**: (a) 単独起動は既存コード上で本番・テスト双方で使用されていない、CLAUDE.md alpha ポリシー「非推奨パスは保持せず削除」。

### Q7: テスト戦略
**決定**: 新規コンポーネントに単体テスト + 既存テスト全 PASS 維持。
**根拠**: (a) CLAUDE.md「テストを通すためにテストコードを修正してはならない」、R6 の外部挙動不変要件の検証線として既存テスト群を用いる。

## 将来拡張

- 登録インターフェースは本 spec では本体利用のみだが、プリセットや将来機能が独自サービスを追加する拡張点として残す。
- Phase2 以降の新サービス追加時も、本 spec で定義する単一の取得インターフェースを使用する方針とする。
