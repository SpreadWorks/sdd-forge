## SDD (Spec-Driven Development)

This project uses Spec-Driven Development powered by sdd-forge.

### About docs/

`docs/` is a structured knowledge base covering the project's design, architecture, and business logic.
Read docs to understand the full picture before making changes.

**If docs and source code conflict, source code is the truth.**

docs content consists of 3 types:

| Type | Source | Description |
|---|---|---|
| Analysis data | `{{data}}` directives | Structured information auto-extracted from source code |
| AI-generated text | `{{text}}` directives | Explanatory text generated by AI from source and context |
| User-provided info | User input | Operational details not available from source code, written outside directives |

#### Freshness check

Before starting work, compare modification dates of docs/ and source code.
If source is newer, suggest running `sdd-forge build` to the user.

### SDD Flow

When asked to add or modify features, run the `/sdd-flow-start` skill.
If skills are unavailable, follow these steps in order.

**Principle: Confirm the next action with the user at every step. Never proceed without asking.**

1. Confirm approach with user
2. Confirm branch strategy with user
3. `sdd-forge spec --title "<feature-name>"` to create spec (use existing if available)
4. (If requirements gathering chosen) Draft phase (see AI behavior rules)
5. Present spec summary, confirm with user
6. After approval, update `spec.md` `## User Confirmation`
7. `sdd-forge gate --spec specs/NNN-xxx/spec.md`
8. If gate FAILs, resolve open questions one by one
9. Test phase (see AI behavior rules)
10. Implement
11. `sdd-forge forge --prompt "<summary of changes>" --spec specs/NNN-xxx/spec.md`
12. `sdd-forge review`

**User-facing options:**

Step 1 — Approach:
1. **Gather requirements**
2. **Create spec directly**

Step 2 — Branch strategy:
1. **Branch**
2. **Worktree**
3. **Spec only**

Step 5 — Spec review:
1. **Implement**
2. **Modify spec**
3. **Other**

**AI behavior rules:**

| Step | Rule |
|---|---|
| 2 | Inside a worktree: auto-select `--no-branch` (no user confirmation needed) |
| 4 | Record in `specs/NNN-xxx/draft.md`. Ask ONE question at a time (no batching, no self-answering). Digression: resolve in 1 exchange, otherwise record in Open Questions. After approval, transcribe to spec; keep draft.md |
| 5 (modify) | Receive feedback → update spec.md → return to step 5 |
| 8 | Resolve one by one (no implementation allowed) |
| 9 | Auto-detect test env from analysis.json. Exists: present test observations → user approval → generate tests. None: AI performs spec-implementation alignment check. Needs setup: treat as separate spec |

**Failure policy:**
- No implementation until gate PASS
- Not complete until review PASS
- No implementation without user confirmation
- No skipping test observation review when test environment exists

### Closing

After implementation is complete, run the `/sdd-flow-close` skill.
If skills are unavailable, present these numbered options to the user:

1. **Update docs + commit + merge + delete branch**
2. **Update docs**
3. **Commit**
4. **Commit + merge**
5. **Commit + merge + delete branch**

Execute the following based on the user's selection:

| Action | Applies to | Steps |
|---|---|---|
| Update docs | 1, 2 | `sdd-forge forge` → `sdd-forge review` (repeat until PASS) |
| Commit | 1, 3, 4, 5 | `sdd-forge gate --phase post` → commit on feature branch |
| Merge | 1, 4, 5 | Merge to base branch (`config.flow.merge`) → delete `.sdd-forge/current-spec` |
| Delete branch | 1, 5 | Delete feature branch |

### sdd-forge Commands

| Command | Description |
|---|---|
| `sdd-forge help` | Show command list |
| `sdd-forge setup` | Project setup + config generation |
| `sdd-forge build` | Full doc generation (scan → init → data → text → readme) |
| `sdd-forge scan` | Source code analysis → analysis.json |
| `sdd-forge init` | Initialize docs/ from templates |
| `sdd-forge data` | Resolve `{{data}}` directives with analysis data |
| `sdd-forge text --agent <name>` | Resolve `{{text}}` directives with AI |
| `sdd-forge readme` | Auto-generate README.md |
| `sdd-forge forge --prompt "<content>"` | Iterative doc improvement |
| `sdd-forge review` | Doc quality check |
| `sdd-forge changelog` | Generate change_log.md from specs/ |
| `sdd-forge agents` | Update AGENTS.md |
| `sdd-forge spec --title "<name>"` | Initialize spec (feature branch + spec.md) |
| `sdd-forge gate --spec <path>` | Spec gate check (`--phase pre/post`) |
| `sdd-forge flow --request "<request>"` | Auto-run SDD flow |
| `sdd-forge snapshot <save\|check\|update>` | Snapshot testing (regression detection) |

### docs/ Editing Rules

- docs/ content is primarily auto-generated from source code analysis
- Content inside `{{data}}` / `{{text}}` directives is overwritten by auto-generation
- Content outside directives is preserved
- Chapter ordering is defined by the `chapters` array in `preset.json`
- After implementation, run forge → review per the SDD flow
